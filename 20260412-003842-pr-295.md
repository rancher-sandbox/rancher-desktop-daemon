## Executive Summary

This PR introduces the engine controller that mirrors Docker state into Kubernetes resources, adds a "wait for desired state" default to `rdd set`, ships BATS coverage for the new flow, and adds a CI support-bundle wrapper that captures diagnostics when BATS targets hang. The design cleanly separates user intent (`spec.state`) from engine reality (`status.status`), and the finalizer/mirror contract is sensibly documented. However, the implementation ships one critical regression (volume name sanitization allows characters that K8s rejects, crashing the entire full sync on common volume names), plus eight important issues clustered around finalizer error handling (dropped K8s resources on Docker errors, silently-swallowed Conflict errors, edge-triggered cleanup that can't retry), containerd backend support, the semantics of the new wait logic, and per-reconcile O(N) list sweeps that multiply under event load.

Structure: 1 critical, 8 important, 8 suggestions.

## Critical Issues

C1. **`sanitizeKubernetesObjectName` leaves volume names that fail RFC 1123 validation, breaking the entire fullSync** â€” `pkg/controllers/app/engine/controllers/sync_images.go:27-29`, `pkg/controllers/app/engine/controllers/sync_volumes.go:66-72` [Gemini 3.1 Pro]

```go
func sanitizeKubernetesObjectName(input string) string {
	return strings.NewReplacer("/", "-", ":", ".").Replace(input)
}
```

Docker permits underscores and uppercase letters in volume names (e.g., `My_Volume`, `project_db_data`, any Compose-created volume), but `sanitizeKubernetesObjectName` only replaces `/` and `:`. When `applyVolume` calls `w.k8s.Apply(...)` with a name containing `_` or `A-Z`, the Kubernetes API server rejects it due to RFC 1123 subdomain validation on `metadata.name`.

The failure is not localized. `syncAllVolumes` collects errors from each `applyVolume` and returns `errors.Join(errs...)` at `pkg/controllers/app/engine/controllers/sync_volumes.go:89`; `fullSync` at `pkg/controllers/app/engine/controllers/docker_watcher.go:316-329` joins and returns that to `newDockerWatcher`, which aborts the watcher entirely. One volume with an uppercase letter or underscore means the engine never connects, `ContainerEngineReady` stays `False`, and `rdd set running=true` hangs for its full 5-minute timeout.

Fix: produce a deterministic encoding that always yields a valid RFC 1123 subdomain â€” e.g. `fmt.Sprintf("vol-%x", sha256.Sum256([]byte(vol.Name)))` â€” and keep the original Docker name in `status.name` (which `deleteVolume` already uses for the reverse path via `v.Status.Name`). The same treatment is needed for image tags since they can contain uppercase characters as well.

## Important Issues

I1. **Engine controller unconditionally targets Docker, so containerd backends never satisfy `ContainerEngineReady`** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:76-107`, `pkg/controllers/app/engine/controllers/docker_watcher.go:42-55` [Codex GPT 5.4, Gemini 3.1 Pro]

```go
running := meta.IsStatusConditionTrue(app.Status.Conditions, "Running")
...
if running && !watcherRunning {
	log.Info("App is running, starting Docker watcher")
	if err := r.startWatcher(ctx); err != nil {
		log.Error(err, "Failed to start Docker watcher")
		_ = r.setEngineCondition(ctx, &app, metav1.ConditionFalse, "ConnectFailed", err.Error())
		return ctrl.Result{}, err
	}
```

The engine reconciler decides the watcher lifecycle purely from the App's `Running` condition; it never consults `app.Spec.ContainerEngine.Name`. Meanwhile `lima-template.yaml:47-52` explicitly disables `docker.service` when `PARAM_CONTAINER_ENGINE=containerd`, and `newDockerWatcher` unconditionally pings the Docker socket. So for a containerd-backed VM, the reconciler enters an endless `ConnectFailed` loop.

This intersects directly with the new `rdd set` wait path: the help text at `cmd/rdd/set.go:70-72` advertises `rdd set running=true containerEngine.name=containerd` as a supported example, and `waitForDesiredState` at `cmd/rdd/set.go:450-454` waits for `ContainerEngineReady=True`. In containerd mode that condition can never become True, so the command always hangs until the 300-second timeout.

Fix: gate the watcher on `app.Spec.ContainerEngine.Name == "moby"` (or any other Docker-compatible engine you add later). For containerd, either short-circuit `ContainerEngineReady` to a distinct terminal reason (e.g. `Unsupported` / `NotApplicable`) so the CLI can detect "nothing to wait for," or skip the condition entirely and drive the CLI off the App's `Running` condition instead. Either way, the "waiting for container engine" branch in `waitForDesiredState` needs a matching adjustment.

I2. **`ContainerNamespace` mirror finalizer is never processed on user delete â€” objects stick in `Terminating` until the next VM shutdown** â€” `pkg/controllers/app/engine/controllers/sync_volumes.go:44-50`, `pkg/controllers/app/engine/controllers/engine_reconciler.go:266-281`, `pkg/controllers/app/engine/controllers/engine_reconciler.go:354-361` [Codex GPT 5.4]

```go

	applyConfig := containersv1alpha1apply.ContainerNamespace(containerNamespace, apiNamespace).
		WithFinalizers(mirrorFinalizer)

	return w.k8s.Apply(ctx, applyConfig,
		client.ForceOwnership, client.FieldOwner(controllerName))
}
```

`syncContainerNamespace` now adds the `engine.rancherdesktop.io/docker-mirror` finalizer to the `moby` ContainerNamespace.

`processFinalizers` at `engine_reconciler.go:266-281` only covers Containers, Images, and Volumes, and `SetupWithManager` at lines 354-361 does not watch ContainerNamespaces.

The existing `ValidateDelete` webhook at `pkg/controllers/containers/containernamespace/container_namespace_webhooks.go:24-29` returns only a warning, so users can still issue `kubectl delete containernamespace/moby`. The object gets a deletion timestamp, retains the mirror finalizer, and is never unstuck until the next VM stop (when `cleanupMirrorResources` strips finalizers wholesale). The pre-existing "namespace delete TODO" just graduated into a visible dead end.

Fix: either stop adding the mirror finalizer to ContainerNamespace at all (it isn't reverse-managed and `cleanupMirrorResources` can still sweep it at stop time), or add a dedicated ContainerNamespace watch + finalizer handler and tighten the delete webhook to reject user-initiated deletes of system-managed namespaces up front.

I3. **Finalizer is removed even when the Docker-side delete fails, silently orphaning the real Docker object** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:283-340`, `pkg/controllers/app/engine/controllers/docker_watcher.go:273-304` [Codex GPT 5.4]

```go
func (r *EngineReconciler) processContainerFinalizers(ctx context.Context, w *dockerWatcher) error {
	...
	for i := range containers.Items {
		c := &containers.Items[i]
		if c.DeletionTimestamp == nil || !hasFinalizer(c, mirrorFinalizer) {
			continue
		}
		w.deleteContainer(ctx, c.Name)
		if removeFinalizer(c, mirrorFinalizer) {
			if err := r.Update(ctx, c); err != nil {
				return fmt.Errorf("failed to remove finalizer from container %s: %w", c.Name, err)
			}
		}
	}
	return nil
}
```

`deleteContainer` / `deleteImage` / `deleteVolume` at `docker_watcher.go:273-304` swallow all Docker errors (`V(1)` log only). The finalizer callers (`processContainerFinalizers`, `processImageFinalizers`, `processVolumeFinalizers`) then unconditionally strip the finalizer and delete the K8s mirror.

This directly violates the contract in `docs/design/api_containers.md:56-59`: "the finalizer handler deletes the corresponding object in the container engine, then removes the finalizer to allow K8s garbage collection." A transient Docker daemon error, permission error, or conflict (container still running, image in use, volume mounted) leaves the real Docker object in place while the Kubernetes mirror is gone for good. Eventual divergence is guaranteed unless the next `fullSync` rebuilds the mirror â€” and a user who tried to delete it via the API will be surprised that it has been silently recreated.

Fix: make the delete helpers return errors. Ignore only the "not found" / already-gone cases (e.g. `errdefs.IsNotFound`), treat all other errors as a failure, and leave the finalizer in place so `processFinalizers` retries on the next reconcile. If the error is persistently terminal (e.g. "container is in use"), surface it on the Container's conditions so the user can see why deletion is blocked.

I4. **`rdd set running=false` wait returns on `ContainerEngineReady` change, not on the VM actually stopping** â€” `cmd/rdd/set.go:436-462` [Codex GPT 5.4]

```go
const cond = "ContainerEngineReady"
if runningVal == "true" {
	logrus.Info("Waiting for container engine to be ready")
	return watchCondition(ctx, config, func(obj *unstructured.Unstructured) bool {
		return conditionStatus(obj, cond) == metav1.ConditionTrue
	})
}
logrus.Info("Waiting for container engine to disconnect")
return watchCondition(ctx, config, func(obj *unstructured.Unstructured) bool {
	return conditionStatus(obj, cond) != metav1.ConditionTrue

```

The help text at `cmd/rdd/set.go:65-67` promises that `running=false` "waits for the VM to stop", but `waitForDesiredState` returns as soon as `ContainerEngineReady` is absent or `False`. That condition is already `False` on the engine controller's `ConnectFailed` path, can be `False` while the App controller is still propagating `running=false` to the LimaVM, and can flip to `False` long before the LimaVM actually finishes shutting down. In those windows the command reports success while the VM is still running.

Fix: for `running=false`, wait on the App's `Running` condition going to `False` (or the LimaVM's equivalent stopped condition). Reserve `ContainerEngineReady` for the `running=true` / Docker branch. This also sidesteps the I1 containerd hang on the `running=true` side once the gating there is fixed.

I5. **Cleanup on VM stop is fragile end-to-end: no Conflict retry, first-error short-circuit, and an edge-triggered outer guard that makes retries impossible** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:78-126`, `pkg/controllers/app/engine/controllers/engine_reconciler.go:210-234` [Claude Opus 4.6, Gemini 3.1 Pro]

```go
for _, item := range items {
	obj := item.(client.Object)
	if removeFinalizer(obj, mirrorFinalizer) {
		if err := r.Update(ctx, obj); err != nil {          // no retry on Conflict
			return fmt.Errorf("failed to remove finalizer from %s/%s: %w", ...)
		}
	}
	if err := client.IgnoreNotFound(r.Delete(ctx, obj)); err != nil {
		return fmt.Errorf("failed to delete %s/%s: %w", ...)
	}
}
```

Three stacked weaknesses turn any transient problem during cleanup into lasting damage:

1. **Inner loop has no Conflict retry** [Claude Opus 4.6]. `deleteAllOfType` calls `r.Update(ctx, obj)` to strip finalizers with no `retry.RetryOnConflict` wrapper. A stale cache or concurrent modification during shutdown returns `Conflict`, which propagates up as a generic error.
2. **First error short-circuits the remaining types** [Claude Opus 4.6]. `cleanupMirrorResources` at `engine_reconciler.go:190-207` calls `deleteAllOfType` four times in sequence for Container / Volume / Image / ContainerNamespace. Any error in the first call returns before the remaining three run, leaving a half-cleaned state across kinds.
3. **Outer retry is edge-triggered on in-memory watcher state** [Gemini 3.1 Pro]. `stopWatcher()` sets `r.watcher = nil` *before* `cleanupMirrorResources` runs at `engine_reconciler.go:115-124`. When cleanup returns an error the reconcile requeues â€” but on the retry, `watcherRunning` is now `false`, the `!running && watcherRunning` branch is skipped entirely, and no further cleanup attempt happens until the user stops the VM again.

A successful next start partially self-heals via the `syncAllContainers` / `syncAllImages` / `syncAllVolumes` stale-item pruning, so this is not permanent orphaning â€” but between stop and that next start, the resources are visible to `rdd ctl get` and violate the BATS assertion at `bats/tests/32-app-controllers/engine.bats:160-168`. Objects mid-deletion (finalizer already stripped, Delete not yet issued) can also land in a broken state.

Fix: (1) wrap the inner `Update` in `retry.RetryOnConflict` with the default backoff; (2) collect errors across the four types with `errors.Join` instead of returning on the first; (3) decouple "cleanup needed" from the in-memory watcher pointer â€” run cleanup whenever `!running` regardless of `watcherRunning`, or persist a short-lived "cleanup pending" condition on the App.

I6. **Initial `fullSync` issues N sequential `Inspect` calls per container/image, amplifying cold-start latency** â€” `pkg/controllers/app/engine/controllers/sync_containers.go:39-45`, `pkg/controllers/app/engine/controllers/sync_images.go:45-55` [Gemini 3.1 Pro]

```go
for _, dc := range listResult.Items {
	dockerIDs[dc.ID] = true
	if err := w.syncContainer(ctx, dc.ID); err != nil {
		errs = append(errs, err)
	}
}
```

`syncAllContainers` calls `ContainerList`, then loops and calls `ContainerInspect` once per container. `syncAllImages` does the same with `ImageList` + `ImageInspect`. A developer laptop with several hundred images (not uncommon on a long-lived machine) hits the Docker socket hundreds of times sequentially before the watcher goroutine enters its event loop, and every hit goes through the Lima port-forwarded Unix socket. Observed startup latency for `rdd set running=true` will grow linearly with the image count, and the operation is entirely serial â€” `newDockerWatcher` holds `watcherMu` the whole time.

This also intersects with I1: even in the correct Docker case, any developer with a bloated local cache pays this cost on every VM restart.

Fix: seed the initial mirror from `List` responses where the shape is sufficient (image list has `RepoTags`, `Size`, `Labels`, etc.), and defer `Inspect` to event-driven updates or lazy per-resource fetches when a user actually queries a field that only `Inspect` exposes. If an Inspect is still needed for full status, do it concurrently with a small worker pool so startup isn't bounded by serial round-trips.

I7. **Every App reconcile re-lists Containers, Images, and Volumes; per-event work scales O(N) with mirror size** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:128-140`, `pkg/controllers/app/engine/controllers/engine_reconciler.go:266-341`, `pkg/controllers/app/engine/controllers/engine_reconciler.go:354-361` [Claude Opus 4.6, Codex GPT 5.4]

```go
// Handle container spec changes (state transitions).
if running {
	if err := r.reconcileContainerSpecs(ctx); err != nil {
		return ctrl.Result{}, fmt.Errorf("failed to reconcile container specs: %w", err)
	}
}

// Handle finalizer processing for resources being deleted.
if running {
	if err := r.processFinalizers(ctx); err != nil {
		return ctrl.Result{}, fmt.Errorf("failed to process finalizers: %w", err)
	}
}
```

Every reconcile with `running=true` executes four unfiltered list operations:

- `List(ContainerList)` in `reconcileContainerSpecs` at line 247
- `List(ContainerList)` in `processContainerFinalizers` at line 285
- `List(ImageList)` in `processImageFinalizers` at line 305
- `List(VolumeList)` in `processVolumeFinalizers` at line 325

The reconcile fires on *every* Container / Image / Volume watch event via `Watches(&containersv1alpha1.Container{}, enqueueApp)` at `engine_reconciler.go:358-360`, and each status update the Docker watcher writes back generates one such event. So every container status change on a host with N mirrored containers performs 3Â·N K8s reads and a nested `reconcileContainerState` sweep that calls `ContainerStart` / `ContainerStop` against Docker for every container with a non-`unknown` spec. On a busy system with dozens of containers cycling the multiplier is immediate and user-visible.

The same architecture also made it easy to miss `ContainerNamespace` coverage entirely (see I2) â€” the App-level sweep approach has no compile-time guarantee that every mirrored kind has a finalizer handler.

Fix: split the engine controller into (a) an App-level session manager that only owns the Docker watcher lifecycle and (b) per-resource reconcilers (`Container`, `Image`, `Volume`, `ContainerNamespace`) that handle their own finalizer and spec updates with `For()` predicates filtering to "deletion timestamp set" or "spec generation changed." That removes the O(N) sweeps and makes coverage explicit per kind.

I8. **`removeMirrorResource` uses `IgnoreNotFound` on `Update`, silently swallowing `Conflict` / `Forbidden` / every other error** â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:225-236` [Claude Opus 4.6]

```go
if removeFinalizer(obj, mirrorFinalizer) {
	if err := w.k8s.Update(ctx, obj); err != nil {
		return client.IgnoreNotFound(err)     // swallows Conflict/Forbidden/etc.
	}
}
return client.IgnoreNotFound(w.k8s.Delete(ctx, obj))
```

`client.IgnoreNotFound` returns `nil` only for `IsNotFound` errors and passes everything else through â€” but here it is being used as a generic "swallow if it fails" on `Update`, which is wrong. `Conflict` on the finalizer-strip `Update` (the common optimistic-concurrency outcome when two goroutines race on the same mirror object) is silently treated as success, and the subsequent `Delete` then runs against a stale snapshot. The only expected benign error on `Update` here is `NotFound` (object deleted out from under us); everything else should surface.

This path is reachable for every Docker `Destroy` event through `handleContainerEvent` / `handleImageEvent` / `handleVolumeEvent`, which makes it a frequent trigger in normal use. Combined with I5 above, a Conflict here leads to a finalizer that was never actually removed plus a Delete that appears to succeed locally â€” exactly the kind of divergence the finalizer contract is meant to prevent.

Fix: drop `IgnoreNotFound` on the `Update` and either (a) propagate the error so the event loop retries, or (b) wrap in `retry.RetryOnConflict` with a bounded backoff and only propagate on terminal errors. Keep `IgnoreNotFound` on the final `Delete` â€” that one is correct.

## Suggestions

S1. **Design doc still documents `ContainerCreateRequest.spec.state` as defaulting to `running`** â€” `docs/design/api_containers.md:156-160` [Codex GPT 5.4]

```yaml
spec:
  name: magical_gates
  namespace: k8s.io
  state: running # Default to `running`
```

The API default in `pkg/apis/containers/v1alpha1/container_types.go:217-220` and the generated CRD at `pkg/controllers/containers/container/crd.yaml:133` both say `default: unknown`. Consumers reading the design doc will get the wrong startup semantics for create requests.

Fix: update the example to `state: unknown` (and drop the `Default to running` comment), or, if create-requests really should auto-start, change the API default back to `running` and document why the Container default is different.

S2. **`applyContainer` only sets the finalizer on first create; a manually stripped finalizer never gets restored** â€” `pkg/controllers/app/engine/controllers/sync_containers.go:86-97` [Gemini 3.1 Pro]

```go
var existing containersv1alpha1.Container
err := w.k8s.Get(ctx, client.ObjectKey{Name: inspect.ID, Namespace: apiNamespace}, &existing)
if apierrors.IsNotFound(err) {
	applyConfig := containersv1alpha1apply.Container(inspect.ID, apiNamespace).
		WithFinalizers(mirrorFinalizer).
		WithSpec(containersv1alpha1apply.ContainerSpec().WithState(containersv1alpha1.ContainerStatusUnknown))
	if err := w.k8s.Apply(ctx, applyConfig, client.ForceOwnership, client.FieldOwner(controllerName)); err != nil {
		return fmt.Errorf("failed to create container %s: %w", inspect.ID, err)
	}
}
```

`applyContainer` deliberately only writes `spec` on first create so that later syncs do not clobber a user-initiated `spec.state`. That's correct for the spec, but as a side effect it also means the mirror finalizer is only written once â€” if it is ever missing on a subsequent sync (manually stripped, or a new API server behavior), it never gets restored. Since Apply supports setting only specific fields, the structural metadata (finalizers, labels) can be applied unconditionally on every sync without touching `spec.state`.

Fix: split the apply into two field-owner scopes: one that always writes finalizers and any owner-managed metadata, and one that only writes `spec.state=unknown` on the not-found path.

S3. **`rdd set` watch loop aborts on any watch-channel closure instead of reconnecting** â€” `cmd/rdd/set.go:496-499` [Gemini 3.1 Pro]

```go
case event, ok := <-watcher.ResultChan():
	if !ok {
		return errors.New("watch channel closed while waiting for App state")
	}
```

Kubernetes watches routinely close for benign reasons: API server timeouts, proxies cutting idle connections, resource-version compaction. Returning a hard error on closure means any sufficiently long-running `rdd set` may fail even though the desired state would eventually be reached. The `rdd set` timeout is 300s by default, which is already in range for a typical watch disconnect.

Fix: wrap the watch in a retry loop, ideally via `k8s.io/client-go/tools/watch` (`watchtools.UntilWithSync` or similar), so a benign channel close resumes from the last observed resource version.

S4. **`reconcileContainerState` ignores intermediate container states (`paused`, `restarting`), so `spec.state=created` silently no-ops on them** â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:254-268` [Claude Opus 4.6]

```go
case containersv1alpha1.ContainerStatusCreated:
	if actual == containersv1alpha1.ContainerStatusRunning {
		log.Info("Stopping container", "id", c.Name)
		_, err := w.cli.ContainerStop(ctx, c.Name, dockerclient.ContainerStopOptions{})
		return err
	}
```

The `desired=created` branch only stops the container when `actual == ContainerStatusRunning`. If a user patches `spec.state=created` on a container that happens to be in `paused` or `restarting` state, the reconciler returns `nil` and waits for a `running` status it may never see â€” the user's intent is dropped silently. Docker's `ContainerStop` handles paused containers natively.

Fix: drop the narrow `actual == running` guard and call `ContainerStop` for any non-`created`, non-`exited` / non-`dead` state (or just unconditionally and let Docker short-circuit).

S5. **`bats-with-timeout.sh` hand-rolls `instance.LogDir()` logic, with a comment acknowledging the duplication** â€” `scripts/bats-with-timeout.sh:22-28` vs `pkg/instance/instance.go:67-82` [Claude Opus 4.6]

```bash
instance_name="rancher-desktop-${RDD_INSTANCE:-2}"
case "$(uname -s)" in
    Linux)   log_dir="${HOME}/.local/state/${instance_name}" ;;
    Darwin)  log_dir="${HOME}/Library/Logs/${instance_name}" ;;
    MINGW*|MSYS*|CYGWIN*) log_dir="${LOCALAPPDATA:-${HOME}/AppData/Local}/${instance_name}-logs" ;;
    *)       log_dir="${HOME}/${instance_name}-logs" ;;
esac
```

The shell script duplicates the OS â†’ log-dir mapping because `rdd` itself may be hung at timeout time. That reason is legitimate â€” but the two implementations will drift.

Fix: expose the log-dir path as a lightweight helper that doesn't touch the control plane (e.g., `rdd debug log-dir` or a standalone `rdd-log-dir` printing binary) so the shell script can call it even when the service is hung. As a cheaper alternative, add a CI test that asserts the shell mapping matches `instance.LogDir()` across all three platforms.

S6. **Support-bundle output is capped at 30â€“50 lines via `head`, which silently truncates the exact diagnostic detail we'd want after a hang** â€” `scripts/bats-with-timeout.sh:66-67`, `scripts/bats-with-timeout.sh:86-87`, `scripts/bats-with-timeout.sh:98-107` [Claude Opus 4.6]

```bash
lsof -p "$pid" 2>/dev/null | head -30 | sed 's/^/  /' || true
...
ss -tupn 2>&1 | head -50 || true
```

The bundle already exists as a post-mortem artifact rather than a live log stream, so the reason to cap output is unclear â€” and the cap hides the specific file descriptors or sockets that would identify the cause of a hang. `head -30` on `lsof -p` of a busy process is almost certain to drop the interesting fd.

Fix: remove the `head` caps, or hoist them to a configurable `BATS_SUPPORT_BUNDLE_MAX_LINES` that defaults high (or to 0 for unlimited) and stays low only on explicit opt-in.

S7. **Add a comment explaining why `applyContainer` uses a Get-then-Apply pattern instead of pure server-side Apply** â€” `pkg/controllers/app/engine/controllers/sync_containers.go:86-97` [Claude Opus 4.6]

The Get-then-Apply pattern is correct: server-side Apply with field ownership would *still* reassert `spec.state=unknown` on every sync (a field the engine controller does not own after creation), which would clobber user-initiated `spec.state=running` / `spec.state=created`. The Get check ensures `WithSpec` is only called on the not-found path.

Fix: a two-line comment above the `Get` call explaining the invariant â€” "spec.state must be written exactly once, on creation; subsequent syncs must not reassert it to avoid clobbering user intent." Saves the next reader from puzzling out why the Apply omits `WithSpec` unconditionally.

S8. **Dead `scheme` field in `dockerWatcher`** â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:30`, `pkg/controllers/app/engine/controllers/docker_watcher.go:41`, `pkg/controllers/app/engine/controllers/docker_watcher.go:63` [Claude Opus 4.6]

```go

	cli    *dockerclient.Client
	k8s    client.Client
	scheme *runtime.Scheme // never read
	...
}
```

`scheme *runtime.Scheme` is plumbed through `newDockerWatcher`, stored on the struct, and never read anywhere in the package (verified with `grep -n 'w\.scheme'`). Remove the field and propagate the removal up through `EngineReconciler.startWatcher` at `engine_reconciler.go:154` and the controller wiring at `pkg/controllers/app/engine/controller.go:59`.

## Design Observations

**Concerns**

- **Docker-only coupling in shared APIs** *(in-scope)* [Gemini 3.1 Pro] â€” The `Container`, `Image`, `Volume`, and `ContainerNamespace` types live under `pkg/apis/containers`, implying general container-engine support, but the engine controller is deeply coupled to Docker (hardcoded `moby` namespace, direct moby SDK). ```

`cleanupMirrorResources` (`engine_reconciler.go:190-207`) calls `deleteAllOfType` four times in sequence; any error in the first type short-circuits the remaining three. Combined with the non-retrying optimistic `Update` on line 223, a stale cache or concurrent-modification `Conflict` during shutdown leaves containers half-cleaned while volumes/images/namespaces remain. Self-heals on next startup via `fullSync` in `docker_watcher.go:309`, but between stop and start the stale resources are observable via `rdd ctl get` and violate the promise of the bats assertion at `bats/tests/32-app-controllers/engine.bats:160-168`. Wrap `Update` in `retry.RetryOnConflict` and collect errors across types instead of returning on the first.

**I2 â€” `removeMirrorResource` masks non-`NotFound` Update errors**
`pkg/controllers/app/engine/controllers/docker_watcher.go:225-236`

```go
if removeFinalizer(obj, mirrorFinalizer) {
    if err := w.k8s.Update(ctx, obj); err != nil {
        return client.IgnoreNotFound(err)     // swallows Conflict/Forbidden/etc.
    }
}
return client.IgnoreNotFound(w.k8s.Delete(ctx, obj))
```

`IgnoreNotFound` is the wrong filter here; the only expected benign error on `Update` is `NotFound`. A `Conflict` on the finalizer-strip update is silently treated as success, and the subsequent `Delete` runs against a stale snapshot. Either fail loudly (and let the caller retry) or specifically retry on Conflict. This is reachable from `handleContainerEvent`/`handleVolumeEvent` for every Docker `Destroy` event.

**I3 â€” `processFinalizers` + `reconcileContainerSpecs` run on every reconcile, O(N) each**
`pkg/controllers/app/engine/controllers/engine_reconciler.go:128-140, 266-341`

When `running=true` each reconcile unconditionally performs:
- 1 Ă— `List(ContainerList)` in `reconcileContainerSpecs` (line 247)
- 1 Ă— `List(ContainerList)` in `processContainerFinalizers` (line 285)
- 1 Ă— `List(ImageList)` in `processImageFinalizers` (line 305)
- 1 Ă— `List(VolumeList)` in `processVolumeFinalizers` (line 325)

Every status update from the Docker watcher enqueues a reconcile via `Watches(&containersv1alpha1.Container{}, enqueueApp)` at line 358. So each container status change triggers four List calls (plus a nested per-container `reconcileContainerState` pass that calls `ContainerStart`/`ContainerStop`). On a busy system with dozens of containers cycling, this is effectively O(N) work per status update and multiplies rapidly. Predicates (e.g., filter for `DeletionTimestamp != nil` updates for finalizers; filter for `spec.State` generation changes for state reconciliation) would cut this by an order of magnitude.

### SUGGESTIONS

**S1 â€” `reconcileContainerState` ignores intermediate states**
`pkg/controllers/app/engine/controllers/docker_watcher.go:254-268`

The `desired=created` branch only stops when `actual == ContainerStatusRunning`. If a user patches `spec.state=created` on a container in `paused`/`restarting` state, nothing happens â€” the reconciler waits for the container to reach `running` which it may never do. Docker's `ContainerStop` handles paused containers natively; drop the narrow `actual == running` guard.

**S2 â€” `err2` variable name**
`pkg/controllers/app/engine/controllers/sync_images.go:81-82`

The surrounding function already used `err` at line 76 in a scope that has ended; there's no shadowing concern. Rename to `err` and fold into a single return.

**S3 â€” `bats-with-timeout.sh` duplicates `instance.LogDir()` logic**
`scripts/bats-with-timeout.sh:22-28` vs `pkg/instance/instance.go:67-82`

The shell script hand-rolls the OSâ†’log_dir mapping to avoid calling `rdd` (which may be hung). The comment at `bats-with-timeout.sh:20-21` already acknowledges the duplication. Consider a Go helper that prints just `LogDir()` without touching any runtime state, invoked as `rdd print log-dir` or similar â€” callable even when the control plane is stuck. Alternatively, add a test that asserts both implementations produce the same path.

**S4 â€” Goroutine watcher recovery depends on a non-obvious status-update re-trigger**
`pkg/controllers/app/engine/controllers/engine_reconciler.go:88-100, 345-362`

When the watcher errors, the reconciler cleans up and sets `ContainerEngineReady=Disconnected`, then returns `nil`. Recovery relies on the `Status().Update()` at line 186 triggering another reconcile through `For(&appv1alpha1.App{})`. This works in practice, but the single-buffer `reconcileChan` (line 345) means if a second event is dropped by the non-blocking send in `enqueueReconcile` (line 134-139), the only remaining trigger is the status update. Add a test for "Docker engine restarts while app is running" to lock this in.

**S5 â€” Stale-detection is O(N*M) per full sync**
`sync_containers.go:47-60`, `sync_images.go:56-69`, `sync_volumes.go:74-87`

Each full sync lists all K8s resources and for each checks membership in the `activeNames`/`dockerIDs` map â€” fine, already O(N) with the map. Good. But note the sync itself issues one `ContainerInspect`/`ImageInspect` per item in serial (`sync_containers.go:42`, `sync_images.go:46`). For hundreds of images, the initial sync serializes hundreds of HTTP calls â€” worth a parallelism bump (`errgroup`) if sync time matters.

**S6 â€” `dump_sockets` and `dump_macos_ps` output lines are capped at 30â€“50 via `head`, which will silently truncate in noisy environments**
`scripts/bats-with-timeout.sh:66-67, 86-87, 98-106`

Diagnostic bundles benefit from being complete; the cap seems designed to keep the bundle small but will hide the exact file descriptors or processes that caused the hang. Consider removing the `head` caps or making them configurable, since this is a post-mortem artifact, not a log stream.

**S7 â€” `applyContainer` Get-then-Apply pattern is correct but expensive**
`pkg/controllers/app/engine/controllers/sync_containers.go:86-97`

Fetching the container before applying is the right mechanism to avoid overwriting user-set `spec.state` â€” Server-Side Apply would merge based on field ownership, but the initial spec needs to be created exactly once with `spec.state=unknown` and never reasserted. The logic is correct. Consider adding a short comment explaining *why* the Get is needed (the apply-config intentionally omits `WithSpec` after creation), since the reason is not obvious from the code alone.

## Design Observations

- **Mirror pattern consistency**: `apiNamespace = "rancher-desktop"` and `containerNamespace = "moby"` hardcoded in `engine_reconciler.go:37-38` match the mock controller at `pkg/controllers/mock/mock_controller.go:27-29`. This is intentional and not a regression â€” the two mirror-style controllers share the same namespace convention, distinct from the app-controller pattern of `app.GetResourceNamespace()` (which returns `"default"` by design per `pkg/apis/app/v1alpha1/app_types.go:94-99`). Worth a comment or doc note so the next reader doesn't treat it as inconsistency.
- **Webhook interaction**: Verified the Container webhook only intercepts `Update` (`container_controller.go:88-90`) and ContainerNamespace only intercepts `Delete` (`container_namespace_controller.go:79-81`). The engine controller's `Apply` (Create semantics) and `Delete` paths therefore bypass the "create not implemented" error. Good catch on the design, but fragile â€” any future registration change would break the engine silently.
- **Spec normalization trick in webhook**: `container_webhooks.go:49-50` copies the old spec, overwrites `State` with the new value, then compares. Clean and correct â€” rejects any change where the diff is not purely `spec.state`.
- **`ContainerStatusUnknown` default**: Changing the default from `running` to `unknown` in `container_types.go:59-60` plus the enum addition is the key enabler: the reconciler ignores `unknown`, so Docker-originated containers don't get state-corrected by accident. Consistent with the doc at `api_containers.md:127-139`.
- **No Windows path**: `engine.bats:20` uses `skip_on_windows`, `DockerSocket()` in `instance.go:129-137` uses unix domain path, and the watcher hardcodes `"unix://" + socketPath` in `docker_watcher.go:44`. WSL2 support is deliberately deferred.
