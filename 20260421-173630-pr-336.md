## Executive Summary

This PR unifies the `--wait` / `--timeout` surface across `rdd svc` and `rdd limavm`, and routes deadline expiry through `cliexit.Timeout` so timeouts exit with code 4 (matching `rdd set`). The refactor is clean and the wrap-once-at-the-CLI-boundary pattern is right. Two behavior changes warrant attention before merging: `svc delete --wait=false` now lets `service.Delete()` proceed to `os.RemoveAll(instance.Dir())` while the serve subprocess is still holding PID/SQLite/log handles; and the shared `defaultWaitTimeout = 30s` is below both the service's own 45s controller-manager drain (`svc stop`/`svc delete`) and the previous 90s CRD-establishment budget (`svc start`, still 90s inside `ensureServiceRunning`). Documentation in `docs/design/cmd_lima.md` was not updated to match the new defaults, the new exit-code-4 contract is asserted in exactly one test, and the wrapper `service.Stop()` removal is unmentioned in the commit body.

## Important Issues

I1. **`svc delete --wait=false` removes the instance directory while the serve subprocess is still alive** — `pkg/service/cmd/service.go:424-438`, `cmd/rdd/service.go:347-376` [Claude Opus 4.7] [Codex GPT 5.4] [Gemini 3.1 Pro]

```go
func Delete(wait bool, timeout time.Duration) error {
    if !Exists() {
        return fmt.Errorf("%q control plane does not exist", instance.Name())
    }
    if Running() {
        if err := StopWithWait(wait, timeout); err != nil {
            return err
        }
    }
    preserveAllInstanceLogs()
    if os.Getenv("RDD_KEEP_LOGS") == "" {
        _ = os.RemoveAll(instance.LogDir())
    }
    _ = os.RemoveAll(instance.ShortDir())
    return os.RemoveAll(instance.Dir())
}
```

Before the PR, `Delete()` always called `Stop()`, which always waited. This PR exposes `--wait=false` on `svc delete` and threads it through to `StopWithWait`. When the user passes `--wait=false`, `StopWithWait` signals the process and returns immediately (`pkg/service/cmd/service.go:358-395`); `Delete` then calls `os.RemoveAll(instance.LogDir())`, `os.RemoveAll(instance.ShortDir())`, and `os.RemoveAll(instance.Dir())` while the serve subprocess still holds open handles to `rdd.pid`, `rdd.sqlite3`, `rdd.stderr.log`, `rdd.stdout.log`, and has its working directory pinned to `instance.Dir()` (set in `service.go:607`).

On Windows, `os.RemoveAll` fails with a sharing violation mid-removal because the handles are still open; the user sees an error and a partially-deleted tree. On Unix, removal succeeds, but then `Running()` starts returning false (PID file is gone; `PID()` returns `PIDNotFound` at `pkg/service/cmd/service.go:138-161`) even though the serve process remains alive. A subsequent `rdd svc start` for the same instance now sees `!Exists()`, recreates the directory, and starts a second serve subprocess that races the first for the secure port (`6443 + instance.Index()`).

Gemini flagged this as critical; Codex and Claude as important. We consolidate at important: the data being removed is itself destined for deletion, but the PID-file-based mutual exclusion breaks and the Windows partial-deletion path leaves the user with an error.

Fix: always wait for the process to exit before deleting state. Either drop `--wait=false` from `svc delete` entirely (deletion cannot usefully be non-blocking since the writer must be gone) or pass `true` unconditionally to the internal `StopWithWait` call while still exposing `--timeout` so users can tune how long to wait:

```diff
     if Running() {
-        if err := StopWithWait(wait, timeout); err != nil {
+        // Data deletion requires the writer to be gone before the directory
+        // is removed; --wait=false on svc delete is not safe, so ignore it.
+        if err := StopWithWait(true, timeout); err != nil {
             return err
         }
     }
```

If `--wait=false` is preserved for delete, `docs/design/cmd_service.md:112` should document the race explicitly.

I2. **Shared `defaultWaitTimeout = 30s` is below the service's own graceful-shutdown budget and below the previous start-readiness budget** — `cmd/rdd/service.go:32-34`, `cmd/rdd/service.go:234, 343, 354`, `pkg/service/cmd/service.go:815-827` [Claude Opus 4.7] [Codex GPT 5.4] [Gemini 3.1 Pro]

```go
// defaultWaitTimeout is the default --timeout for subcommands whose wait
// normally settles within seconds.
const defaultWaitTimeout = 30 * time.Second
```

The new default wires into three subcommands with different sizing constraints:

*Stop / delete path.* The serve subprocess allows up to 45s for its controller-manager goroutine to drain (`pkg/service/cmd/service.go:826`: `case <-time.After(45 * time.Second)`), and Lima hostagents get 30s of that budget for graceful shutdown. When the CLI's 30s deadline fires first, `StopWithWait` calls `process.Kill(pid)` at line 384, aborting the subprocess's own 45s window. The now-removed comment inside `StopWithWait` captured exactly this: `// The service needs up to 30s for graceful hostagent shutdown plus overhead, so allow 60s total.` With the new default, a healthy shutdown of a service running a Lima VM will routinely force-kill instead of draining cleanly.

*Start path.* The replaced `WaitWithTimeout` used 90s with the comment `// Increased timeout for CRD establishment`. Both `ensureServiceRunning` call sites (`cmd/rdd/service.go:97, 104`) still use 90s, which is the correct budget for CRD establishment + controller-manager registration + discovery ConfigMap readiness. The explicit `rdd svc start` path at line 234 now uses 30s. Either 90s is right in both places or 30s is right in both places — the split is itself evidence that 30s is too tight for the readiness path.

*Test exposure.* `bats/tests/20-service/2-create.bats:65`, `4-start-parameters.bats:45-63`, and `5-port-fallback.bats:72, 105, 147` all run `run -0 rdd svc stop` against services that may have a live Lima VM. Those now risk exit-code-4 flakes when the hostagent drain exceeds 30s.

Fix: split the one constant into named constants that each match a concrete invariant. For example:

```diff
-// defaultWaitTimeout is the default --timeout for subcommands whose wait
-// normally settles within seconds.
-const defaultWaitTimeout = 30 * time.Second
+// startWaitTimeout covers CRD establishment plus controller-manager registration.
+const startWaitTimeout = 90 * time.Second
+
+// stopWaitTimeout must exceed the service's own 45s controller-manager drain
+// (see Run in pkg/service/cmd/service.go) plus overhead.
+const stopWaitTimeout = 60 * time.Second
```

Then use `startWaitTimeout` in both `newServiceStartCommand` and `ensureServiceRunning`, and `stopWaitTimeout` in `newServiceStopCommand` and `newServiceDeleteCommand`.

## Design Observations

### Concerns

- **One `defaultWaitTimeout` collapses three distinct sizing constraints** (in-scope) [Claude Opus 4.7] [Codex GPT 5.4]. `svc start` covers CRD establishment + controller-manager registration, `svc stop` covers the serve subprocess's own 45s controller-manager drain, and `svc delete` covers stop-then-cleanup. Those invariants have different right answers; the PR uses one 30s constant for all three and still needs a separate 5-minute `limaLongWaitTimeout` for VM boot. Three named constants — each tied to an invariant in its doc comment — would make tuning any one of them safer. See I2.
- **`svc delete` mixes process-shutdown with durable-state removal** (in-scope) [Claude Opus 4.7] [Codex GPT 5.4]. The `--wait=false` footgun in I1 is structural: deleting `instance.Dir()` while the writer still owns it cannot be made safe with better timeouts or doc warnings. The simplest design is "delete always waits for the process to exit." If a non-blocking delete is needed, it needs a two-phase design (e.g., a "pending deletion" marker) that defers filesystem cleanup until after the daemon is confirmed dead.

### Strengths

- `errors.Is(err, context.DeadlineExceeded)` plus `fmt.Errorf("…%w", err)` is the right classification primitive. It traverses the wrap chain so both the `limavm.go` wrap and the `service.go` wrap flow through `errors.As(err, &cliexit.Error)` in `main.go` without a sentinel string match. [Claude Opus 4.7]
- `watchtools.ContextWithOptionalTimeout` is used consistently at every wait site, so the "timeout=0 → no deadline" semantics live in one place instead of six independent timer constructions. [Claude Opus 4.7]
- `StopWithWait` now blocks on `<-ctx.Done()` instead of `<-time.After(60 * time.Second)`, so a parent-context cancellation (e.g., Ctrl-C) actually interrupts the wait. [Claude Opus 4.7]
- Routing `cliexit.Timeout` through a small helper at the CLI boundary (`timeoutError`) keeps `cliexit` concerns out of the lower-level service package. [Codex GPT 5.4]

# Deep Review — PR #336

## Executive Summary

This PR standardizes `--wait` and `--timeout` flags across `rdd svc` and `rdd limavm` subcommands and routes deadline expiry through `cliexit.Timeout` so timeouts exit with code 4 (matching `rdd set`). The refactor is mostly clean, but the new defaults shorten previously-hardcoded timeouts in two paths (60s→30s for `svc stop`, 90s→30s for `svc start` already-running branch), and the new `--wait=false` semantics for `svc delete` introduce a race where the instance directory is removed while the service is still running.

## Findings

### Critical Issues

None.

### Important Issues

**I1. `svc stop --timeout=30s` default is below the service's own ~45s shutdown ceiling — `cmd/rdd/service.go:343`, `pkg/service/cmd/service.go:338-393`** *(important, regression)*

```go
// cmd/rdd/service.go:342-343
command.Flags().Bool("wait", true, "Wait for control plane to actually stop")
command.Flags().Duration("timeout", defaultWaitTimeout, "Timeout for --wait (0 means wait indefinitely)")
```

`defaultWaitTimeout` is `30 * time.Second` (line 34). The previous `StopWithWait` used a hardcoded 60s for exactly this scenario, and the now-removed comment explained why:

> "Wait for the process to actually terminate. The service needs up to 30s for graceful hostagent shutdown plus overhead, so allow 60s total."

The serve subprocess itself waits up to 45s for the controller manager goroutine to drain (`pkg/service/cmd/service.go:826`: `case <-time.After(45 * time.Second):`). With a 30s client-side timeout, `StopWithWait` will reach `<-ctx.Done()` and call `process.Kill(pid)` (line 384), aborting the subprocess's own 45s graceful window. Hostagents that would have completed shutdown in time are now SIGTERM'd by the parent kill — exactly the symptom the previous 60s default was sized to avoid. CI suites that rely on `run -0 rdd svc stop` (`bats/tests/20-service/2-create.bats:65`, `4-start-parameters.bats:45-63`, `5-port-fallback.bats:72`) will see a new flake mode: exit code 4 instead of 0, with leftover hostagents.

Fix: keep 60s (or longer) for `svc stop`, since the existing comment captures a sizing constraint the new default no longer satisfies.

```diff
-// defaultWaitTimeout is the default --timeout for subcommands whose wait
-// normally settles within seconds.
-const defaultWaitTimeout = 30 * time.Second
+// defaultWaitTimeout applies to start/delete waits that normally settle in
+// seconds.
+const defaultWaitTimeout = 30 * time.Second
+
+// stopWaitTimeout matches the service's own 45s controller-manager drain
+// (pkg/service/cmd/service.go Run) plus overhead. The serve subprocess will
+// SIGKILL itself if the manager is still running after 45s, so a 60s client
+// budget covers the worst case without orphaning hostagents.
+const stopWaitTimeout = 60 * time.Second
```

…and use `stopWaitTimeout` for `svc stop` and `svc delete`.

**I2. `svc delete --wait=false` removes the instance directory while the serve process is still running — `cmd/rdd/service.go:358-376`, `pkg/service/cmd/service.go:424-439`** *(important, regression)*

```go
// pkg/service/cmd/service.go:424-432
func Delete(wait bool, timeout time.Duration) error {
    if !Exists() {
        return fmt.Errorf("%q control plane does not exist", instance.Name())
    }
    if Running() {
        if err := StopWithWait(wait, timeout); err != nil {
            return err
        }
    }
    preserveAllInstanceLogs()
    ...
    return os.RemoveAll(instance.Dir())
}
```

Previously `Delete` always called `Stop()`, which always waited (`return StopWithWait(true)`). The new `--wait=false` path on `svc delete` calls `StopWithWait(false, ...)`, which sends a single SIGINT and returns without waiting. `Delete` then proceeds to `os.RemoveAll(instance.LogDir())` and `os.RemoveAll(instance.Dir())` while the serve subprocess is still alive and has open handles to `rdd.pid`, `rdd.sqlite3`, `rdd.stderr.log`, `rdd.stdout.log`, plus its CWD pinned to `instance.Dir()` (set at `service.go:607`).

- On Windows, `os.RemoveAll` returns an access-denied error mid-removal because those handles are still open; the user gets a partial deletion plus an error, and the still-running service keeps writing to a half-removed tree.
- On Unix the removal succeeds, but the now-orphan service keeps writing to deleted FDs. Worse, since `instance.PIDFile()` is gone, `Running()` returns false on the next invocation (`PID()` returns `PIDNotFound` when the PIDFile is unreadable) — `rdd svc start` will happily spawn a *second* serve subprocess for the same RDD_INSTANCE while the original is still alive, since the PIDFile-based mutual exclusion is gone.

## Design Observations

### Concerns

- **Single `defaultWaitTimeout` collapses three distinct sizing constraints — *(in-scope)*.** `svc start` has to cover CRD establishment + controller-manager registration (~tens of seconds in CI), `svc stop` has to cover the serve subprocess's own 45 s controller-manager drain, and `svc delete` has to cover stop-then-cleanup. They legitimately have *different* right answers; the PR collapses them into one 30 s constant for ergonomic uniformity, then has to make `limavm` defaults a separate 5-minute constant anyway. If you keep three named constants (`startReadyTimeout`, `stopShutdownTimeout`, `limaLongWaitTimeout`), each can be sized to the actual underlying ceiling and the comment can name the invariant. This is the change implied by I1+I3.

- **`--wait=false` on `svc delete` is structurally unsafe — *(in-scope)*.** As I2 documents, removing `instance.Dir()` while the writer still holds it is a footgun on every OS. Rather than papering over it with a longer timeout or a doc warning, the cleanest design is: `svc delete` must wait for the process to exit before it removes anything. `--timeout` still tunes that wait, but `--wait=false` for delete simply doesn't make sense — it would be "asynchronously corrupt my data store."

### Strengths

- Routing deadline expiry through `errors.Is(err, context.DeadlineExceeded)` (rather than a sentinel string match) is the right call; the wrapping in `fmt.Errorf("...%w", err)` propagates correctly through both layers (`limavm.go` and `service.go`).
- `watchtools.ContextWithOptionalTimeout` for the `timeout=0 → no deadline` semantics is a small thing but it's the right primitive — every wait site uses it consistently, so a future reviewer can grep one helper instead of auditing six independent timer constructions.
- `StopWithWait` correctly switches from `time.After(60 * time.Second)` to `<-ctx.Done()`, which means the wait can now be cancelled by the parent context (a Ctrl-C during a long stop). The previous code blocked the goroutine until the 60 s timer fired regardless of the user's signal.
