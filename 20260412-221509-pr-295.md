## Executive Summary

Round 5 of PR #295 lands the engine controller that mirrors Docker `Container`/`Image`/`Volume` state into the existing `containers.rancherdesktop.io` API group, plus a new default-wait contract for `rdd set`, CI support-bundle improvements, and related Lima template fixes. The watcher/reconciler split is well-structured, the SSA field-manager workaround is carefully documented, and the BATS coverage for the happy paths is solid.

The review found **one critical regression** and **six important issues**, all correctability without architectural change:

- **Critical (C1):** `rdd set` hangs for the full timeout when the App is already running, because the engine reconciler skips `setEngineCondition` on the steady-state path and `ContainerEngineReady.observedGeneration` stays behind the post-patch generation.
- **Important (I1):** `ContainerNamespace/moby` is not watched, so a user delete stays gone until the next VM restart â€” and the new BATS test cements that broken behaviour.
- **Important (I2):** `processImageFinalizers`/`processVolumeFinalizers` skip the `retry.RetryOnConflict` pattern that `processContainerFinalizers` uses, so a single 409 wastes a Docker delete on the next reconcile.
- **Important (I3):** The new `rdd set` wait predicates treat `StartFailed`/`ConnectFailed` as "keep waiting" and `StopFailed` as "success," so the default wait is misleading on both failure paths.
- **Important (I4):** Docker `ActionRename` is not in the event switch, so `docker rename` leaves `status.name` stale until another state event refreshes the container.
- **Important (I5):** A transient Docker `Inspect` failure during `syncAllImages` causes the valid K8s `Image` mirror to be classified stale and deleted.
- **Important (I6):** The watcher run loop logs and drops errors from `handleEvent`, so a transient apiserver 503 can permanently desynchronise an image or volume mirror until the next VM restart.

Nine suggestions cover SSA list-ordering churn, lock-while-closing asymmetry in `Reconcile`, `DockerSocket()` side effects, error-message specificity, and a few dead/unreachable branches.

Structure: **1 critical, 6 important, 9 suggestions.**

## Critical Issues

C1. **`rdd set` hangs for the full timeout when the App is already running** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:82-192` [Gemini 3.1 Pro]

```go

        // ...
        return ctrl.Result{}, err
    }
    if err := r.setEngineCondition(ctx, &app, metav1.ConditionTrue, "Connected", "Container engine synced"); err != nil {
        return ctrl.Result{}, err
    }
    // Fall through so pending spec.state patches and finalizers ...
}

// reconcileContainerSpecs + processFinalizers ... (no setEngineCondition call)
```

`rdd set` now waits by default (commit `762c1e8`). The wait predicate at `cmd/rdd/set.go:478-481` looks for `ContainerEngineReady.status == True && observedGeneration >= minGen`, where `minGen` is `app.Generation` right after `patchApp` (`cmd/rdd/set.go:425-447`). That generation filter is the exact fix for stale-snapshot races and is correct by itself.

The engine reconciler, however, only calls `setEngineCondition` in two branches: the `!wantWatcher` terminal-state branch (lines 124-155) and the `!watcherRunning` fresh-start branch (lines 157-175). In the steady-state path â€” App running, watcher already running â€” the reconciler falls through directly to `reconcileContainerSpecs`/`processFinalizers` and returns without touching `App.Status.Conditions`. Because `meta.SetStatusCondition` is also only called in those two branches, `ContainerEngineReady.observedGeneration` keeps its previous value and never advances past whatever generation was in effect when the watcher last started.

Concretely, any command that bumps `app.Generation` *without* also cycling the watcher hangs for the full 5 minute default:

```
$ rdd set running=true kubernetes.enabled=true   # when already running=true
# patchApp â†’ spec changes â†’ generation N â†’ N+1
# waitForDesiredState â†’ minGen=N+1
# Reconciler reruns but takes the steady-state path â†’ observedGen stays N
# watchCondition loops until timeout â†’ "timed out waiting for App state"
```

The hang is not limited to that one permutation â€” any property change that leaves `running=true` and doesn't restart the watcher hits it (e.g., toggling `kubernetes.*`, `containerEngine.mirrors`, etc., along with an idempotent `running=true`). The simple start/stop BATS flow in `engine.bats` doesn't exercise it because it only toggles `running` through terminal states, which is why the regression slipped through.

Fix: move the `setEngineCondition(Connected)` call out of the `!watcherRunning` block so every `wantWatcher=true` reconcile re-asserts the condition. `setEngineCondition` already no-ops when nothing changed (`meta.SetStatusCondition` compares all fields) and wraps the write in `retry.RetryOnConflict`, so an unconditional call is cheap on stable reconciles but stamps the current `latest.Generation` into `observedGeneration` whenever the App spec advances.

```diff

 		log.Info("App is running, starting Docker watcher")
 		if err := r.startWatcher(ctx); err != nil {
 			log.Error(err, "Failed to start Docker watcher")
 			if condErr := r.setEngineCondition(ctx, &app, metav1.ConditionFalse, "ConnectFailed", err.Error()); condErr != nil {
 				log.Error(condErr, "Failed to update ContainerEngineReady to ConnectFailed")
 			}
 			return ctrl.Result{}, err
 		}
-		if err := r.setEngineCondition(ctx, &app, metav1.ConditionTrue, "Connected", "Container engine synced"); err != nil {
-			return ctrl.Result{}, err
-		}
 		// Fall through so pending spec.state patches and finalizers ...
 	}
 
+	if err := r.setEngineCondition(ctx, &app, metav1.ConditionTrue, "Connected", "Container engine synced"); err != nil {
+		return ctrl.Result{}, err
+	}
+
 	if err := r.reconcileContainerSpecs(ctx); err != nil {
```

A matching BATS test should run `rdd set running=true <some-unrelated-prop>=<value>` from the running state and assert that it returns well before the 5-minute timeout. Classification: regression, critical.

## Important Issues

I1. **`ContainerNamespace/moby` is not re-mirrored after user delete** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:495-513` [Codex GPT 5.4] [Claude Opus 4.6]

```go
return ctrl.NewControllerManagedBy(mgr).
    For(&appv1alpha1.App{}).
    Named("engine-reconciler").
    WatchesRawSource(source.Channel(r.reconcileChan, enqueueApp)).
    Watches(&containersv1alpha1.Container{}, enqueueApp).
    Watches(&containersv1alpha1.Image{}, enqueueApp).
    Watches(&containersv1alpha1.Volume{}, enqueueApp).
    Complete(r)
```

`syncContainerNamespace()` at `pkg/controllers/app/engine/controllers/sync_volumes.go:62-66` creates `containernamespace/moby` exactly once from `fullSync` on watcher startup. The reconciler watches `Container`, `Image`, and `Volume` but not `ContainerNamespace`, and `ContainerNamespace` has no mirror finalizer, so a user who runs `rdd ctl delete containernamespace/moby` succeeds and produces no event that wakes the engine reconciler back up. Subsequent Container/Image/Volume Docker events trigger reconciles, but `reconcileContainerSpecs`/`processFinalizers` never touch the namespace â€” it stays gone until the VM is restarted (or the watcher dies and fullSyncs).

The design doc at `docs/design/api_containers.md:88-96` says the dockerd namespace "cannot be modified in any way," but nothing currently enforces that. Worse, the new BATS test at `bats/tests/32-app-controllers/engine.bats:364-370` asserts that deleting `containernamespace/moby` leaves it absent. That locks in the broken invariant instead of protecting it, and any future fix will also need to update this test.

Fix: either add `Watches(&containersv1alpha1.ContainerNamespace{}, enqueueApp)` to `SetupWithManager` and idempotently re-run `ensureNamespace + syncContainerNamespace` on every reconcile (both are single Applies and already idempotent), or reject deletion of the `moby` namespace via an admission webhook while the engine controller is active. The watch-and-restore variant is the smaller change and consistent with the existing mirror pattern. Either way, update the BATS test to assert the namespace returns â€” not that it stays gone. Classification: regression, important.

I2. **`processImageFinalizers`/`processVolumeFinalizers` skip `retry.RetryOnConflict`** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:431-492` [Gemini 3.1 Pro] [Claude Opus 4.6]


`processContainerFinalizers` (lines 410-423) correctly re-Gets `latest` under `retry.RetryOnConflict` before stripping the finalizer. The Image and Volume helpers mutate the stale `img`/`v` from the outer List and call `Update` directly. On a 409 â€” which can fire any time a concurrent watcher sync just re-applied status â€” the Update fails and the finalizer isn't stripped. The Docker-side delete has already succeeded by that point, so the reconciler requeues, re-Lists, re-inspects, and re-issues the engine-side delete against an already-gone object before finally stripping the finalizer. It's self-healing but burns a Docker round-trip per conflict and breaks the invariant "once we tell Docker to delete, we strip the finalizer on the first attempt," which the adjacent Container path upholds.

Fix: mirror the `processContainerFinalizers` pattern in both helpers. Put the finalizer strip inside a `retry.RetryOnConflict` loop that re-Gets `latest` and treats `NotFound` as benign:

```diff

-            if err := client.IgnoreNotFound(r.Update(ctx, img)); err != nil {
-                errs = append(errs, fmt.Errorf("failed to remove finalizer from Image %s: %w", img.Name, err))
-            }
-        }
+        key := client.ObjectKeyFromObject(img)
+        retryErr := retry.RetryOnConflict(retry.DefaultRetry, func() error {
+            latest := &containersv1alpha1.Image{}
+            if err := r.Get(ctx, key, latest); err != nil {
+                if apierrors.IsNotFound(err) {
+                    return nil
+                }
+                return err
+            }
+            if !removeFinalizer(latest, mirrorFinalizer) {
+                return nil
+            }
+            return r.Update(ctx, latest)
+        })
+        if retryErr != nil {
+            errs = append(errs, fmt.Errorf("failed to remove finalizer from Image %s: %w", img.Name, retryErr))
+        }
```

Apply the same change to `processVolumeFinalizers` (lines 485-489). Classification: regression, important.

I3. **`rdd set` wait misinterprets `StartFailed`/`ConnectFailed`/`StopFailed` conditions** â€” `cmd/rdd/set.go:476-494` [Codex GPT 5.4]

```go

    return watchCondition(ctx, config, func(obj *unstructured.Unstructured) bool {
        status, gen, present := conditionInfo(obj, "ContainerEngineReady")
        return present && gen >= minGen && status == metav1.ConditionTrue
    })
}

    if !present {
        return true
    }
    return gen >= minGen && status != metav1.ConditionTrue

```

The `running=true` branch only accepts `ContainerEngineReady=True`. If startup fails â€” LimaVM publishes `Running=False/StartFailed` at `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:382-385,427-430,463-476`, or the engine reconciler publishes `ContainerEngineReady=False/ConnectFailed` at `engine_reconciler.go:161-163` â€” the wait keeps looping until the 5-minute timeout and returns the generic "timed out waiting for App state" error instead of the actual failure reason. Users see a timeout when they should have seen "start failed" immediately.

The `running=false` branch has the opposite problem: it treats *any* refreshed non-`True` `Running` condition as success. When Lima publishes `Running=False/StopFailed` at `limavm_lifecycle.go:527-539`, the wait returns success even though the VM didn't stop.

```

running=true failure   : Running.reason in {StartFailed} OR ContainerEngineReady.reason in {ConnectFailed}
running=false success  : Running absent OR Running.reason in {Stopped}
running=false failure  : Running.reason in {StopFailed}
```

On failure the wait should return an error whose message carries the condition's `message` field so the user sees the underlying cause. Extending `conditionInfo` to also return `reason`/`message` is a small change that keeps `watchCondition` generic. Classification: regression, important.

I6. **Docker watcher drops events on transient `handleEvent` errors** â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:171-199` [Gemini 3.1 Pro]

```go
case msg, ok := <-result.Messages:
    if !ok {
        // closed stream handling ...
    }
    if err := w.handleEvent(ctx, msg); err != nil {
        log.Error(err, "Failed to handle Docker event",
            "type", msg.Type, "action", msg.Action, "actor", msg.Actor.ID)
    }
```

A transient failure inside `handleEvent` â€” apiserver 503, SSA conflict that survives its internal retry, temporary network partition â€” is logged and dropped. The watcher never periodically resyncs (there's no time-based `fullSync` tick), so whether the mirror eventually catches up depends on whether Docker emits *another* event for the same object:

- **Container:** a subsequent Start/Stop/Die usually arrives, so drift tends to self-heal.
- **Image:** a pull event fires once. If its apply fails, the `Image` mirror never shows up until the next watcher restart or manual pull of the same tag.
- **Volume:** a create event also fires once. Same permanent-drift risk.

Gemini's proposed fix â€” return from `run` on any error and let the dead watcher be detected by the reconciler â€” is correct but aggressive: every transient 503 would tear down the event stream, trigger a fresh `fullSync`, and churn every mirror. A gentler fix is enough in practice:

1. On `handleEvent` error, enqueue an event-specific retry (e.g., store `msg.Actor.ID` in a small retry set and call `syncContainer`/`reconcileImageByID`/`syncVolume` on a short backoff), **or**
2. Run `fullSync` on a periodic tick (e.g., every 5 minutes) as a belt-and-suspenders resync, so any missed event is reconciled within a bounded window.

Option 2 is the smaller change, matches how controller-runtime's `SyncPeriod` works, and is how `kubelet`'s image manager handles the same class of bug. Classification: gap, important.

### Concerns

**D1. `reconcileContainerSpecs` + `processFinalizers` fan out on every mirror event (in-scope).** [Claude Opus 4.6] The existing comment at `engine_reconciler.go:177-183` spells out the cost: every Container/Image/Volume watch event triggers a reconcile that Lists every mirrored object to check for spec.state changes and pending finalizers. ### Strengths

**S1. **I1. `processImageFinalizers`/`processVolumeFinalizers` don't retry-on-conflict and write stale objects** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:431-492`

```go

    if err := client.IgnoreNotFound(r.Update(ctx, img)); err != nil {

```

Unlike `processContainerFinalizers` (engine_reconciler.go:410-423) which re-Gets under `retry.RetryOnConflict` before stripping the finalizer, both `processImageFinalizers` (line 454-458) and `processVolumeFinalizers` (line 485-489) mutate the stale `img`/`v` from the list and Update directly. On a 409 Conflict â€” which happens any time another writer (e.g., an engine watcher sync that just re-applied status) touches the object between our List and Update â€” the Update fails. The Docker-side delete has already succeeded by that point, so the next reconcile will retry `deleteImage`/`deleteVolume` from scratch (wasting an engine round-trip) before attempting the finalizer strip again. -            }
```diff
-        }
+        key := client.ObjectKeyFromObject(img)
+        retryErr := retry.RetryOnConflict(retry.DefaultRetry, func() error {
+            latest := &containersv1alpha1.Image{}
+            if err := r.Get(ctx, key, latest); err != nil {
+                if apierrors.IsNotFound(err) {
+                    return nil
+                }
+                return err
+            }
+            if !removeFinalizer(latest, mirrorFinalizer) {
+                return nil
+            }
+            return r.Update(ctx, latest)
+        })
+        if retryErr != nil {
```

## Design Observations

### Concerns

**D1. `reconcileContainerSpecs` + `processFinalizers` fan out on every mirror event (in-scope, comment-acknowledged).** The existing comment at `engine_reconciler.go:177-183` accurately describes the cost â€” every Container/Image/Volume watch event triggers a reconcile, which in turn Lists every mirrored object to check for spec.state changes and pending finalizers. For large container counts this is O(NÂ˛) across a sync burst. The long-term fix the comment proposes (per-resource reconcilers with watch predicates for `deletionTimestamp!=nil` and `spec.state` changes) is correct. The near-term impact is bounded because typical dev machines carry few containers, but this will need attention before the feature supports larger workloads. Classification: future.

**D2. Two field managers ("engine-controller" and "engine-controller-finalizer") to work around SSA spec ownership (in-scope).** The explanation at `sync_containers.go:144-157` is correct but it points at a design wart: server-side apply couldn't express "update only finalizers" without releasing spec from the original owner, so a second field manager was introduced. An alternative â€” keep SSA only for the creation path and switch to a strategic merge patch for finalizer re-assertion â€” would avoid the two-manager split and match how most controllers add finalizers. The current approach works; flag for future simplification. Classification: future.

### Executive Summary

This PR introduces the engine controller to mirror Docker's `Container`, `Image`, and `Volume` state into Kubernetes resources. The review identifies a critical bug where `rdd set` hangs because condition generation updates are bypassed for already-running instances. It also flags important data integrity issues on transient errors: dropped Docker events can permanently desynchronize state, and failed image inspects erroneously delete valid K8s mirrors.

### Findings

#### Critical Issues

**C1. `rdd set` hangs because `ContainerEngineReady` `ObservedGeneration` is not updated when the watcher is already running** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:121-167`

If the App is already running and the Docker watcher is active, updating a property (e.g. `rdd set running=true kubernetes.version=1.31.0`) increments the `App`'s `metadata.generation`. `waitForDesiredState` then loops, waiting for `ContainerEngineReady` to reach this new `minGen`. However, because `wantWatcher` and `watcherRunning` are both true, `EngineReconciler` falls through and bypasses the `setEngineCondition` call entirely. The condition's `ObservedGeneration` remains stale, and `rdd set` hangs until its 5-minute timeout.

Fix: Unconditionally assert the `Connected` condition when the watcher is desired and running. `setEngineCondition` already checks for changes, so this is a cheap no-op when stable but ensures `ObservedGeneration` matches the current `App` generation.

```diff
 	if !watcherRunning {
 		log.Info("App is running, starting Docker watcher")
 		if err := r.startWatcher(ctx); err != nil {
 			log.Error(err, "Failed to start Docker watcher")
 			if condErr := r.setEngineCondition(ctx, &app, metav1.ConditionFalse, "ConnectFailed", err.Error()); condErr != nil {
 				log.Error(condErr, "Failed to update ContainerEngineReady to ConnectFailed")
 			}
 			return ctrl.Result{}, err
 		}
-		if err := r.setEngineCondition(ctx, &app, metav1.ConditionTrue, "Connected", "Container engine synced"); err != nil {
-			return ctrl.Result{}, err
-		}
 		// Fall through so pending spec.state patches and finalizers
 		// made during the watcher-down window are processed on this
 		// same reconcile, instead of waiting for a later event. The
 		// restart-after-crash path would otherwise stall if the
 		// ContainerEngineReady condition hadn't changed (setEngineCondition
 		// is a no-op) and no mirror watch event follows.
 	}
 
+	if err := r.setEngineCondition(ctx, &app, metav1.ConditionTrue, "Connected", "Container engine synced"); err != nil {
+		return ctrl.Result{}, err
```

#### Important Issues

**I1. Docker watcher drops events on transient Kubernetes API errors** â€” `pkg/controllers/app/engine/controllers/docker_watcher.go:189-192`

In the `run` loop, if `w.handleEvent(ctx, msg)` fails (e.g., a transient 503 from the API server or a conflict when applying the status update), the error is merely logged and the event is dropped. Because the controller relies entirely on the event stream and does not periodically resync, the Kubernetes mirror becomes permanently out-of-sync with Docker.

Fix: Return from the loop on error. The deferred `recover()` block can be adjusted or the `w.enqueueReconcile()` call can be executed directly before returning. The reconciler will then observe the dead watcher, restart it, and perform a reliable `fullSync`.

```diff
 			if err := w.handleEvent(ctx, msg); err != nil {
-				log.Error(err, "Failed to handle Docker event",
+				log.Error(err, "Failed to handle Docker event; restarting watcher to resync",
 					"type", msg.Type, "action", msg.Action, "actor", msg.Actor.ID)
+				return
 			}
```
*(important, gap)*

**I2. Transient `Inspect` failures in `syncAllImages` cause valid Image mirrors to be erroneously deleted** â€” `pkg/controllers/app/engine/controllers/sync_images.go:59-64`

In `syncAllImages`, `syncImageFromSummary` inspects the image. If this Docker API call fails transiently, it logs the error and continues. The expected mirror names are never added to `activeNames`. The subsequent cleanup loop then erroneously considers the valid Kubernetes `Image` mirrors as stale and deletes them. 

Fix: Derive the expected mirror names from the `summary`'s `RepoTags` directly and add them to `activeNames` *before* calling `syncImageFromSummary`, ensuring transient inspect failures do not trigger mirror deletion.

```diff
 	var errs []error
 	for _, summary := range listResult.Items {
+		if len(summary.RepoTags) > 0 {
+			for _, tag := range summary.RepoTags {
+				activeNames[fmt.Sprintf("%s-%x", sanitizeKubernetesObjectName(summary.ID), sha256.Sum256([]byte(tag)))] = true
+			}
+		} else {
+			activeNames[sanitizeKubernetesObjectName(summary.ID)] = true
+		}
 		names, err := w.syncImageFromSummary(ctx, summary)
 		if err != nil {
 			log.Error(err, "Skipping image during full sync", "id", summary.ID)
 			continue
 		}
-		for _, n := range names {
-			activeNames[n] = true
-		}
 	}
```
*(important, bug)*

**I3. Missing `retry.RetryOnConflict` in `processImageFinalizers` and `processVolumeFinalizers`** â€” `pkg/controllers/app/engine/controllers/engine_reconciler.go:440`

`processContainerFinalizers` rightly wraps finalizer removal in `retry.RetryOnConflict` to survive stale caches and concurrent updates without forcing a whole-reconcile requeue. `processImageFinalizers` and `processVolumeFinalizers` omit this wrapper. A transient conflict on `r.Update` will cause the reconciler to requeue and unnecessarily re-attempt the Docker API `deleteImage`/`deleteVolume` calls.

Fix: Apply the exact same `retry.RetryOnConflict` block used in `processContainerFinalizers` to the Image and Volume finalizer removal logic.

### Design Observations

**Strengths**
- The panic recovery mechanism in `dockerWatcher.run` correctly utilizes the `alive()` check during the `App` reconcile to safely restart a dead watcher and run a `fullSync`. This cleanly prevents permanent lockups or orphaned streams.
- The sorting of port mappings before passing them to the Kubernetes list (`sync_containers.go`) effectively circumvents spurious apply churn that usually plagues `+listType=map` fields.

### Testing Assessment

1. **Wait Condition Timeout (`rdd set`)**: The hanging behavior flagged in C1 lacks test coverage because the BATS tests run `rdd set running=true` after `rdd set running=false`. A test ensuring `rdd set running=true kubernetes.version=xxx` returns successfully when the VM is *already* running would prevent regressions.
2. **Transient API Errors**: There is no coverage for the watcher's ability to recover from dropped events or transient Docker socket timeouts.
