## Executive Summary

## Important Issues

I1. **`ensureServiceRunning` timeouts bypass the `cliexit.Timeout` envelope** — `cmd/rdd/service.go:98-113`, `cmd/rdd/limavm.go:260-280`, `cmd/rdd/set.go:110, 566`, `cmd/rdd/kubectl.go:34` [Codex GPT 5.4] [Gemini 3.1 Pro]

```go
func ensureServiceRunning(ctx context.Context) error {
    if !service.Exists() {
        if err := service.Create(nil); err != nil {
            return err
        }
    }
    if service.Running() {
        ctx, cancel := context.WithTimeout(ctx, startWaitTimeout)
        defer cancel()
        if err := service.Wait(ctx); err != nil {
            return err
        }
        return waitForDiscoveryConfigMap(ctx)
    }
    return startAndWaitForReady(ctx, nil, startWaitTimeout)
}
```

`ensureServiceRunning` now enforces the new `startWaitTimeout` (90s) on every service-autostart path, but it returns raw `context.DeadlineExceeded` without routing through `timeoutError`. Callers — `getKubeClient` in `cmd/rdd/limavm.go:261` (used by every `rdd limavm *` subcommand), `rdd set` at `cmd/rdd/set.go:110` and `:566`, `rdd kubectl` at `cmd/rdd/kubectl.go:34`, and `rdd service config` at `cmd/rdd/service.go:182` — all propagate the error without wrapping. `cmd/rdd/main.go`'s `errors.As(err, &cliexit.Error)` check then sees a bare deadline error and exits with code 1, contradicting the PR's stated uniform exit-code-4 contract. The `svc start` path itself is fine (line 302 wraps `startAndWaitForReady` directly with `timeoutError`), but the shared `ensureServiceRunning` entry point five other commands use is not.

Fix: wrap the error at the source so every caller inherits the contract without a separate `timeoutError` call at each site:

```diff

     if !service.Exists() {
         if err := service.Create(nil); err != nil {
             return err
         }
     }
     if service.Running() {
         ctx, cancel := context.WithTimeout(ctx, startWaitTimeout)
         defer cancel()
-        if err := service.Wait(ctx); err != nil {
-            return err
-        }
-        return waitForDiscoveryConfigMap(ctx)
+        if err := service.Wait(ctx); err != nil {
+            return timeoutError(err)
+        }
+        return timeoutError(waitForDiscoveryConfigMap(ctx))
     }
-    return startAndWaitForReady(ctx, nil, startWaitTimeout)
```

## Suggestions

S1. **`Delete` leaves the directory in place when the process exits between outer and inner `Running()` checks** — `pkg/service/cmd/service.go:338-340, 426-434` [Claude Opus 4.7]

```go
// Delete (outer check)
if Running() {
    if err := StopWithWait(true, timeout); err != nil {
        return err
    }
}
// ...
return os.RemoveAll(instance.Dir())

// StopWithWait (inner check, line 338-340)
func StopWithWait(wait bool, timeout time.Duration) error {
    if !Running() {
        return fmt.Errorf("%q control plane is not running", instance.Name())
    }
```

Pre-PR, `Delete()` called `_ = Stop()` and discarded the error, so a process that exited between any two checks did not block deletion. Post-PR, `Delete(timeout)` propagates the `StopWithWait` error verbatim. A narrow window exists between `Delete`'s outer `Running()` and `StopWithWait`'s inner one in which the serve subprocess can exit (crash, OOM-kill, external `kill`, concurrent `rdd svc stop`). When that happens, `StopWithWait` returns `"… control plane is not running"`, `Delete` propagates it, and `os.RemoveAll(instance.Dir())` never runs. The user sees exit code 1 and a confusing message, even though the desired end state is one `os.RemoveAll` away. The condition self-heals on retry — the next invocation sees `!Running()`, skips the stop branch, and proceeds to delete.

Fix: treat "already not running" as success in `Delete` by re-checking after the stop failure:

```diff
     if Running() {
         if err := StopWithWait(true, timeout); err != nil {
-            return err
+            // The process may have exited between our Running check and
+            // StopWithWait's; in that case its work is already done.
+            if Running() {
+                return err
+            }
         }
     }
```

S3. **`rdd svc start` on an already-running service returns before the `ReadyAnnotation` is set** — `cmd/rdd/service.go:270-278` [Claude Opus 4.7] [Codex GPT 5.4]

```go

    logrus.Infof("%q control plane is already running", instance.Name())
    if !wait {
        return nil
    }
    logrus.Infof("waiting for %q control plane to be ready", instance.Name())
    ctx, cancel := watchtools.ContextWithOptionalTimeout(cmd.Context(), timeout)

```

`docs/design/cmd_service.md:93-95` explicitly advertises the flow "`--wait=false` … then run `rdd service start` again with no options to wait." The cold-start path (`startAndWaitForReady`) waits for both `service.Wait(ctx)` and `waitForFreshDiscoveryConfigMap(ctx, beforeStart)` — the latter requires `ReadyAnnotation == "true"`, which the controller manager sets only after every expected controller has registered. The already-running path calls only `service.Wait(ctx)`, which via `checkReadiness` returns success as soon as the discovery ConfigMap lists registered controllers — before the ready annotation. A concurrent `--wait=false` followed by `rdd svc start` during a still-initializing daemon therefore returns success under a narrower contract than the cold-start path guarantees.

Codex raised this at Important because the PR's new documentation names the workflow explicitly; Claude raised it at Suggestion because the race window is narrow (controllers finish registering quickly, and a steady-state instance is always fine). We consolidate at Suggestion: the practical exposure is a narrow mid-startup race, but the documented workflow deserves a one-line fix to deliver the advertised guarantee.

Fix: mirror `ensureServiceRunning`'s pattern and wait for the discovery ConfigMap on the already-running branch too:

```diff
-    return timeoutError(service.Wait(ctx))
+    if err := service.Wait(ctx); err != nil {
+        return timeoutError(err)
+    }
+    return timeoutError(waitForDiscoveryConfigMap(ctx))
```

## Design Observations

### Concerns

- **Two readiness definitions coexist in `pkg/service/cmd/service.go`** (in-scope) [Codex GPT 5.4]. `service.Wait()` proves API readiness plus best-effort controller discovery; `startAndWaitForReady()` adds the `ReadyAnnotation` + freshness check. Unifying the two into a single readiness helper removes the mismatch behind S3 and gives every future caller the same contract. Matches the shape of S3's fix.
- **`StopWithWait` constructs `context.Background()` instead of accepting the caller's context** (future) [Claude Opus 4.7] [Gemini 3.1 Pro]. The function signature is `StopWithWait(wait bool, timeout time.Duration)` rather than the idiomatic `StopWithWait(ctx context.Context, wait bool, timeout time.Duration)`. Today this is a no-op — `cmd/rdd/main.go:153` passes `context.Background()` and never installs signal-to-context plumbing — but if the CLI ever grows a signal handler that cancels `cmd.Context()`, the shutdown wait will not observe it. Plumbing the context through costs one parameter and matches every other wait site in the CLI.
- **Pre-existing PID-recycling race in `StopWithWait`** (future) [Claude Opus 4.7]. Between `Running()` at line 339 and `pid := PID()` at line 346, the service can exit and `PID()` can find the PID file stale and return `PIDNotFound` (0). `process.Interrupt(0)` on Unix becomes `unix.Kill(0, SIGINT)`, which signals every process in the calling process's process group — including the `rdd` CLI itself and its parent shell. The race is narrow and existed before this PR, but the refactor did not close it.

### Strengths

- **Removing `--wait=false` from `svc delete` is the correct fix for Round 1 I1** [Claude Opus 4.7] [Codex GPT 5.4]. No safe semantics exist for "delete without waiting" when the writer still owns the directory, and `docs/design/cmd_service.md` now names the safety reason explicitly.
- **`startWaitTimeout` and `stopWaitTimeout` each tie their value to a concrete invariant in the doc comment** [Claude Opus 4.7]. `startWaitTimeout` cites CRD establishment and controller-manager registration; `stopWaitTimeout` cites the 45s drain in `Run` and the per-VM shutdown ceiling. Future tuning now anchors on the underlying sizing constraint, not a handwave.
- **`timeoutError` at the CLI boundary plus `errors.Is(err, context.DeadlineExceeded)` is the right classification primitive** [Claude Opus 4.7] [Codex GPT 5.4]. The helper keeps `cliexit` concerns out of the lower-level service package, and the `%w` wrap chain propagates through every layer.
- **`watchtools.ContextWithOptionalTimeout` is used at every wait site** [Claude Opus 4.7]. The "timeout=0 → no deadline" semantics now live in one place instead of six independent timer constructions.
- **`service.Delete(timeout)` is a safer API than the previous `service.Delete()`** [Codex GPT 5.4]. Waiting for shutdown before removing instance state is the right direction for both Windows file-handle semantics and Unix PID-file correctness.

## Review Process Notes

### Skill improvements

- **Add: flag a finding's critical/important severity as suspect when the evidence is a quoted block from a "diff view," not from the file in the worktree.** Severity escalation on purely textual evidence (especially where the claim requires the file to be malformed) should be cross-checked by reading the file at the cited line before accepting the severity. Agents sometimes synthesize a plausible diff and then reason from it as if it were the file; the consolidator must catch this. A reviewer recognises the pattern when the cited text begins with `-`/`+` markers, references "find-and-replace," or claims a syntax-level corruption that would break tooling the PR passed through.

### Repo context updates

- **Add:** When a PR introduces a uniform CLI exit-code contract that applies to "every wait-capable subcommand," enumerate every caller of the shared autostart/autowait helper (e.g., `ensureServiceRunning`) before accepting the PR as "uniform." Cross-cutting contracts that are only wired in at the outermost entry points miss every command that invokes the helper transitively (`rdd set`, every `rdd limavm *`, `rdd kubectl`, `rdd service config` in this PR). Reason: deep-review agents catch the obvious call sites but often miss the transitive ones.

## Executive Summary

This PR unifies the `--wait` / `--timeout` surface across `rdd svc` and `rdd limavm`, routing `context.DeadlineExceeded` through a single `timeoutError` helper that tags the error with `cliexit.Timeout` so deadline expiry exits with code 4 (matching `rdd set`). The follow-up commit `8aa55e4` correctly addresses the round-1 findings: `--wait=false` was removed from `svc delete` (the dangerous "remove `instance.Dir()` under a live serve subprocess" mode is gone), the single 30s `defaultWaitTimeout` was split into `startWaitTimeout` (90s, sized for CRD establishment) and `stopWaitTimeout` (5m, exceeding the service's own 45s controller-manager drain), and the SIGTERM/SIGINT signal-name mismatch in the design doc was fixed. The remaining gaps are tests for the new `svc *` exit-code-4 paths and a small TOCTOU window in the new `Delete` that returns an error if the process exits between the outer `Running()` check and the inner one, leaving the directory in place until the user retries.

**S1. `Delete` returns "control plane is not running" error if the process exits between the outer and inner `Running()` checks** — `pkg/service/cmd/service.go:430-434`, `pkg/service/cmd/service.go:338-340`

```go
// Delete (caller)
if Running() {
    if err := StopWithWait(true, timeout); err != nil {
        return err
    }
}

// StopWithWait (callee), line 338-340:
func StopWithWait(wait bool, timeout time.Duration) error {
    if !Running() {
        return fmt.Errorf("%q control plane is not running", instance.Name())
    }
```

Before this PR, `Delete()` called `_ = Stop()` and discarded the error, so a process that exited between any two checks did not block deletion. After this PR, `Delete(timeout)` propagates the `StopWithWait` error verbatim. There is a small window between `Delete`'s `Running()` check at line 430 and `StopWithWait`'s `Running()` check at line 339 in which the serve subprocess can exit (crash, OOM-killed, manually `kill`'ed, or stopped by another `rdd svc stop`). When that happens, `StopWithWait` returns `"… control plane is not running"`, `Delete` propagates it, and the instance directory is **not** removed. The `cliexit.Timeout` helper in `serviceDeleteAction` does not match this error, so the user sees exit code 1 and a confusing message even though the desired state ("no process, no directory") is exactly one `os.RemoveAll` away.

The condition self-heals on retry — the second `rdd svc delete` invocation sees `!Running()`, skips the stop branch entirely, and proceeds to delete. So this is a UX/observability concern, not data loss.

Fix: treat "already not running" as success in `Delete`. The cleanest version re-checks `Running()` after the failure to distinguish "stop genuinely failed" from "process raced ahead":

```diff
     if Running() {
         if err := StopWithWait(true, timeout); err != nil {
+            // The process may have exited between our Running check and
+            // StopWithWait's; in that case its job is already done.
+            if Running() {
                 return err
+            }
         }
     }
```

## Design Observations

### Strengths

- Wrapping deadline detection in a single `timeoutError(err)` helper at the CLI boundary keeps the `cliexit.Timeout` contract in one place, and `errors.Is(err, context.DeadlineExceeded)` correctly traverses the `fmt.Errorf("…: %w", err)` chain so callers can wrap freely without losing the classification.
- `watchtools.ContextWithOptionalTimeout` is used consistently at every wait site, so the "timeout=0 → no deadline" semantics live in one place.
- `8aa55e4` does the right tuning move: instead of one shared `defaultWaitTimeout` constant, it ties each constant to a concrete invariant in the doc comment (`startWaitTimeout` cites CRD establishment, `stopWaitTimeout` cites the 45s drain in `Run`). Future tuning is anchored to the right thing.
- Removing `--wait=false` from `svc delete` is the correct fix for round-1 I1 — there is no useful "non-blocking delete" with a writer holding the directory, and the doc explicitly explains the safety reason.

### Concerns

- **`StopWithWait` ignores the caller's context** — `pkg/service/cmd/service.go:365-370` constructs `context.Background()` internally. The function signature is `StopWithWait(wait bool, timeout time.Duration)` rather than the idiomatic `StopWithWait(ctx context.Context, wait bool, timeout time.Duration)`. In the current CLI this is a no-op (the parent `cmd.Context()` is itself `context.Background()` because `main.go` does not install signal handling), but if signal handling is ever added to the CLI, the wait will not be Ctrl-C interruptible. Future-proofing this would add a `ctx` parameter; the caller in `cmd/rdd/service.go:333` would then pass `cmd.Context()`. (future)
- **Pre-existing PID-recycling race in `StopWithWait`** — `pkg/service/cmd/service.go:339-362`: `Running()` returns true, then `pid := PID()` reads the file again. If the process exits and the PID file is removed between the two calls, `PID()` returns `PIDNotFound` (0), and `process.Interrupt(0)` on Unix becomes `unix.Kill(0, SIGINT)` — which sends SIGINT to **every process in the calling process's process group**, including the `rdd` CLI itself. Not introduced by this PR (the same race exists in the pre-PR code) but worth a separate fix. (future)

**I1. `StopWithWait` ignores CLI context, blocking SIGINT (Ctrl+C)** — `pkg/service/cmd/service.go:362-385` (important, regression)

```go

		ctx := context.Background()
		if timeout > 0 {
			var cancel context.CancelFunc
			ctx, cancel = context.WithTimeout(ctx, timeout)
			defer cancel()
		}
```

By creating its own `context.Background()`, `StopWithWait` ignores the caller's context (`cmd.Context()`). When a user runs `rdd svc stop --wait` or `rdd svc delete`, pressing Ctrl+C will be ignored by the wait loop, and the CLI will hang unresponsively until the 5-minute timeout expires. As noted in the system guidelines, `context.Background()` must be flagged in new code when a parent context is available.

Furthermore, if the wait loop receives a SIGINT via the caller's context, it should gracefully abort the wait without forcefully killing the background service (`process.Kill(pid)`). `Kill` should only trigger on actual deadline expiry.

Fix: Modify `StopWithWait` and `Delete` to accept the caller's context, wrapping it with the timeout in the CLI layer:
```diff
-func StopWithWait(wait bool, timeout time.Duration) error {
+func StopWithWait(ctx context.Context, wait bool) error {
```
Distinguish between user cancellation and timeout in the select block:
```go
			case <-ctx.Done():
				if errors.Is(ctx.Err(), context.DeadlineExceeded) {
					_ = process.Kill(pid)
				}
				return fmt.Errorf("wait aborted for %q control plane with pid %d: %w", instance.Name(), pid, ctx.Err())
```
In `cmd/rdd/service.go`, wrap `cmd.Context()` with the timeout before calling `StopWithWait` or `Delete`.
