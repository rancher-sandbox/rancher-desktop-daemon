## Design Observations

### Concerns

- **`probeCurrentKubeContext` ignores `ExecProvider`, `AuthProvider`, `CAFile`, and `CertFile`** *(future)* [Gemini 3.1 Pro]. The hand-rolled HTTP client at `kube_reconciler.go:312-332` only honours inline `CAData` / `CertData` / `KeyData` and a bearer token. A developer whose `current-context` uses an EKS exec-credential plugin, a GKE auth-provider, or any file-based cert path will see the probe fail with TLS or 401, and the background goroutine at `kube_reconciler.go:259-263` interprets the failure as "current-context unhealthy" and silently overwrites it with `rancher-desktop`. Verified at `e8033bd` (merge-base) that the construction predates this PR; the new V(1) log lines make the failure mode diagnosable but do not address the auth coverage. A `rest.HTTPClientFor(restCfg)` call would pick up every kubeconfig auth path client-go supports.
- **`removeKubeContext` swallows cleanup errors** *(future)* [Gemini 3.1 Pro]. `kube_reconciler.go:355-374` logs but does not return errors from `clearCurrentKubeContext` and `deleteKubeContext`. The caller at `kube_reconciler.go:83` discards the call and stamps `KubernetesReady=False/NotApplicable`. A Windows file lock or permission error on `~/.kube/config` therefore leaks the `rancher-desktop` entry indefinitely with no retry or backoff. Pre-existing (unchanged from `e8033bd`). Aggregating the errors with `errors.Join` and returning them from `removeKubeContext` would let controller-runtime apply normal backoff.
- **`manageKubeContext` rewrites `~/.kube/config` unconditionally on every reconcile** *(future)* [Gemini 3.1 Pro]. The call at `kube_reconciler.go:128` fires on every reconcile that reaches the healthy branch (17 per the PR's own smoke test). `createReplaceKubeContext` (`kube_context.go:42-88`) re-reads, re-merges, and re-writes the file each time, then spawns a background `probeCurrentKubeContext` goroutine. Pre-existing, but it deserves a deep-equal short-circuit in `createReplaceKubeContext`: skip the write (and the goroutine) when the target cluster, user, and context already match the on-disk values.
- **`cmd/app-controller/main.go` does not blank-import `kubernetes` or `engine`** *(future)* [Claude Opus 4.7]. The standalone development binary at `cmd/app-controller/main.go:9-16` imports only `app` and `demo`, so a developer running `bin/app-controller` exercises neither the engine reconciler nor the kubernetes reconciler this PR diagnoses. Pre-existing (introduced by #356 for `kubernetes` and earlier for `engine`); flagging it here while the file is adjacent.

I1. **`probeCurrentKubeContext` builds an incomplete HTTP client, falsely rejecting complex auth** — `pkg/controllers/app/kubernetes/controllers/kube_reconciler.go:322-351` (important, gap)

```go
	tlsCfg := &tls.Config{ ServerName: restCfg.ServerName }
	if len(restCfg.CAData) > 0 { /* ... */ }
	if len(restCfg.CertData) > 0 && len(restCfg.KeyData) > 0 { /* ... */ }

```

`probeCurrentKubeContext` manually constructs an `http.Client` at lines 322-332 using only inline certificate data (`CAData`, `CertData`, `KeyData`) and a bearer token (line 343). It completely ignores `ExecProvider` (used by AWS CLI/EKS), `AuthProvider` (GCP/GKE), `CAFile`, and `CertFile`. If the user's `current-context` relies on any of these standard mechanisms, the `/healthz` GET at line 345 fails with a TLS error or 401 Unauthorized. Because the probe returns `false` (line 347), the background goroutine in `manageKubeContext` incorrectly assumes the context is unhealthy and silently overwrites the user's active context with `rancher-desktop` (line 259).

Fix: Replace the manual HTTP client construction with upstream client-go helpers. Use `clientcmd.NewNonInteractiveDeferredLoadingClientConfig` with `ExplicitPath` so relative paths resolve correctly against the kubeconfig directory, then use `rest.HTTPClientFor(restCfg)` to get a client that supports all Kubernetes auth providers.

I2. **Silent failure during `removeKubeContext` cleanup leaves stale kubeconfig entries** — `pkg/controllers/app/kubernetes/controllers/kube_reconciler.go:76-80`, `355-374` (important, gap)

```go
	if !app.Spec.Kubernetes.Enabled {
		r.removeKubeContext(ctx)
		return ctrl.Result{}, r.setKubeCondition(...)
	}
```

When Kubernetes is disabled, `Reconcile` triggers cleanup via `r.removeKubeContext(ctx)` at line 76. If `clearCurrentKubeContext` or `deleteKubeContext` fails (e.g., due to a Windows file lock or permission error), `removeKubeContext` only logs the errors at lines 369 and 372 before returning. `Reconcile` then completes with `ctrl.Result{}, nil`. Because the error is swallowed and not returned to the controller-runtime, the reconciler never applies exponential backoff to retry the cleanup. The `rancher-desktop` context permanently leaks in the user's `~/.kube/config`.

Fix: Modify `removeKubeContext` to return an aggregated error (using `errors.Join`) if clearing or deleting the context fails. In `Reconcile` at line 76, check for this error and return it so the framework safely retries the teardown.

`Reconcile` unconditionally calls `manageKubeContext` at line 128 whenever Kubernetes is enabled and running. `manageKubeContext` calls `createReplaceKubeContext` (line 211) which unconditionally locks and rewrites `~/.kube/config` to disk. Because the `App` resource receives frequent status updates from the LimaVM and Engine controllers, `kubernetes-reconciler` triggers often (resulting in the 17 reconcile loops noted in the PR description). ### Design Observations

**Concerns**
- **Aggressive current-context hijacking (in-scope)**: `manageKubeContext` launches a background probe against the user's `current-context`. If the probe fails, it aggressively overwrites their active context with `rancher-desktop`. A developer whose external cluster is temporarily unreachable (e.g., VPN disconnected, sleeping VM) will have their context silently swapped, leading to accidental deployments to the local RDD cluster once they resume work. RDD should only take over the active context if it was already pointing to RDD or if explicitly requested, rather than hijacking it based on reachability.

**Strengths**
- The addition of `AppKubernetesReasonMergeFailed` correctly aligns the controller's behavior with the documented API contract, ensuring that users waiting via `rdd set` are properly blocked on a failed file merge instead of silently hanging.

### Testing Assessment

1. `removeKubeContext` teardown: The `!app.Spec.Kubernetes.Enabled` path that triggers `removeKubeContext` is completely untested. There is no assertion that the `rancher-desktop` entries are actually removed from `~/.kube/config` when the feature is disabled.
2. `probeCurrentKubeContext` edge cases: The probe logic is unexercised by unit tests. The incomplete auth handling (I1) ships green because there are no tests simulating a user's pre-existing complex kubeconfig (like an `ExecProvider` AWS EKS config) during a merge.
3. `probeK3sAPI` transient failures: The "Waiting for k3s API server" (`Probing`) states are not explicitly tested, though they are common in practice during boot.
