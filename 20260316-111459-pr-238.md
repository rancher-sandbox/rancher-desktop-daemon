## Consolidated Review

### Executive Summary

This PR splits the single shared `rdd.rancherdesktop.io/cleanup` finalizer into two purpose-specific finalizers: `/cleanup` for self-protection (a resource's own controller runs cleanup before deletion) and `/owned` for cascade blocking (an owner prevents child deletion until explicit release via `DeleteOwnedResources`). The split fixes a real bug where `DeleteOwnedResources` stripped the only finalizer on a child resource, bypassing the child controller's cleanup logic. The implementation is correct for new resources but leaves existing resources stranded on upgrade: legacy template ConfigMaps still carry `/cleanup`, which `DeleteOwnedResources` no longer strips.

### Important Issues

1. **Legacy template ConfigMaps become stuck in Terminating after upgrade** — `pkg/controllers/base/finalizer.go:159-176` [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

```go
patch := client.MergeFrom(item.DeepCopy())
if controllerutil.RemoveFinalizer(&item, OwnedFinalizerName) {
    if err := c.Patch(ctx, &item, patch); err != nil {
        // ...
    }
}
if err := c.Delete(ctx, &item); err != nil && client.IgnoreNotFound(err) != nil {
    // ...
}
```

Before this PR, the LimaVM webhook created template ConfigMaps with `rdd.rancherdesktop.io/cleanup` (the old `FinalizerName`). After this PR, `DeleteOwnedResources` strips only `/owned`. When a user deletes a LimaVM created before the upgrade:

- `DeleteOwnedResources` tries to strip `/owned` from the template ConfigMap — not present, so no-op.
- `c.Delete` marks the ConfigMap for deletion.
- The old `/cleanup` finalizer blocks actual deletion. No controller removes `/cleanup` from child ConfigMaps — `RemoveCleanupFinalizer` runs only for the resource's own controller.
- The template ConfigMap is stuck in Terminating permanently.

All three agents identified this issue. Claude rated it important/gap; Codex rated it important/regression; Gemini rated it critical/regression. The correct classification is **important, regression**: the PR introduces the incompatibility, but recovery is straightforward in the single-user embedded context (delete the kine database or manually patch the finalizer).

Fix: Migrate legacy children. The simplest approach: in the LimaVM reconciler, when fetching the template ConfigMap during normal reconciliation, check whether it carries `/cleanup` without `/owned`. If so, patch to replace `/cleanup` with `/owned`. This keeps the migration close to the only controller that creates these ConfigMaps and avoids polluting the generic `DeleteOwnedResources` helper with knowledge of the legacy scheme.

### Design Observations

1. **`DeleteOwnedResources` is now schema-version-sensitive** (in-scope) [Codex GPT 5.4]

The helper assumes every child that should be unblocked already uses `/owned`. Encoding a short compatibility window (stripping the legacy finalizer from owned resources) in the helper would eliminate the entire class of upgrade bugs across all controllers, rather than requiring each controller to handle its own migration. The tradeoff: this would contradict the design principle that `DeleteOwnedResources` does not strip `/cleanup`. A targeted migration in the owning controller is safer and keeps the helper's contract clean.

2. **The `/owned` finalizer doubles as a user-facing deletion guard** (in-scope) [Gemini 3.1 Pro]

Child resources like template ConfigMaps do not need `/owned` for the garbage collection loop to work — `DeleteOwnedResources` can delete resources without a finalizer. The `/owned` finalizer's primary value is blocking accidental `kubectl delete` attempts via the `ValidateDelete` webhook. This layered approach (webhook + finalizer) is sound but worth documenting explicitly so future contributors understand the finalizer's dual purpose.

### Design Observations

1. **The owned finalizer could also protect against accidental direct deletion** (in-scope)

   The `ValidateDelete` webhook at `controller.go:276` checks only `HasOwnedFinalizer` to block direct deletion. After `DeleteOwnedResources` strips `/owned`, the webhook allows the delete, and the ConfigMap is deleted. This is correct.

   But if a resource has both `/cleanup` and `/owned`, `DeleteOwnedResources` strips `/owned`, issues a delete, and the resource enters Terminating. The child controller sees `IsBeingDeleted` and runs cleanup, then removes `/cleanup`. The webhook's check is now redundant in this path since the finalizer itself already blocks deletion. The webhook's value is preventing *direct* `kubectl delete` attempts that bypass `DeleteOwnedResources` — a user-facing UX guardrail. This layered approach (webhook + finalizer) is sound.

2. **`DeleteOwnedResources` could strip both owned and legacy finalizers in one pass** (in-scope)

   Rather than adding separate migration logic, `DeleteOwnedResources` could strip *both* `/owned` and `/cleanup` from owned resources. This would handle the upgrade path naturally: any child resource identified via `IsOwnedByUID` gets both stripped, regardless of which was set. The `/cleanup` finalizer on the *owner itself* is untouched (it's never an "item" in the loop). This is simpler than a separate migration step and eliminates the category of upgrade bugs permanently. The tradeoff: if a child resource ever legitimately needs `/cleanup` for its own controller's cleanup, this approach would bypass it. But the design doc already states `DeleteOwnedResources` should *not* strip `/cleanup`, so this would contradict the stated design principle. A targeted migration (strip legacy `/cleanup` only when `/owned` is absent on an owned resource) would be safer.

### Executive Summary

This PR correctly addresses an architectural limitation where a single finalizer forced `DeleteOwnedResources` to short-circuit child resources' self-cleanup logic, successfully splitting responsibilities into `/cleanup` and `/owned` finalizers. However, the change breaks backward compatibility for existing resources (e.g. legacy `TemplateConfigMap`s) that still hold the old `/cleanup` finalizer but rely entirely on `DeleteOwnedResources` for cascade garbage collection.

### Findings

#### Critical Issues

1. **Upgrade compatibility broken for existing `TemplateConfigMap`s** — `pkg/controllers/base/finalizer.go:162-164` `(critical, regression)`

```go
			patch := client.MergeFrom(item.DeepCopy())

```

Before this PR, `TemplateConfigMap`s created by the `LimaVM` mutating webhook were assigned the `rdd.rancherdesktop.io/cleanup` finalizer. With this update, `DeleteOwnedResources` has been modified to exclusively strip `OwnedFinalizerName` (`rdd.rancherdesktop.io/owned`). #### Important Issues

2. **Template ConfigMap orphaned if LimaVM creation fails** — `pkg/controllers/lima/limavm/controller.go:226-236` `(important, gap)`

```go
	// Create the ConfigMap (this triggers ConfigMap admission webhook for validation)
	if err := d.Client.Create(ctx, templateConfigMap, options...); err != nil {
		return fmt.Errorf("failed to create template ConfigMap: %w", err)
	}
```

The mutating webhook generates the `templateConfigMap` and assigns it the `OwnedFinalizerName` finalizer synchronously before the `LimaVM` itself is created in etcd. At this stage, it cannot attach an OwnerReference because the `LimaVM` UID is not yet available. If the `LimaVM` resource creation is subsequently rejected by the API server (e.g., due to a validating webhook, RBAC limits, or an etcd transient error), the ConfigMap is permanently orphaned. Because it lacks an OwnerReference, `DeleteOwnedResources` will never be able to discover and clean it up. Furthermore, because it holds the `/owned` finalizer, the `ConfigMapValidator.ValidateDelete` hook blocks users from manually deleting it.

Fix: Defer the creation of the `templateConfigMap` from the mutating webhook to the `LimaVMReconciler`. Creating the ConfigMap in the reconciler ensures the `LimaVM` UID is available to atomically assign an OwnerReference, preventing side-effect leaks during admission requests.

### Design Observations

**Two Finalizers vs One** `(in-scope)`
The introduction of two distinct finalizers (`/cleanup` for self-protection and `/owned` for cascade blocking) cleanly separates responsibilities. It ensures that `DeleteOwnedResources` does not prematurely short-circuit a child resource's teardown logic. However, this model highlights that child resources not strictly requiring self-protection (like generic `ConfigMap`s) technically do not need an explicit finalizer at all, given RDD completely relies on `DeleteOwnedResources` for manual cascade deletion rather than standard Kubernetes GC. The `/owned` finalizer on these children primarily acts as a user-facing lock to prevent accidental manual deletion, which is useful but not strictly necessary for the garbage collection loop to successfully function.
