## Consolidated Review

### Executive Summary

Three commits make the container `pid` field optional, move mock resources from the `default` to the `rancher-desktop` Kubernetes namespace, and create `ContainerNamespace` resources for all volume namespaces. The new `kubeNamespaceReconciler` has a self-referential ownership bug: it applies `rancher-desktop`'s ownerReference pointing to itself instead of `rdd-mocks`, breaking garbage collection and causing reconciliation oscillation. This should merge with fixes.

**Structure:** 3 commits — make pid optional in container CRD, move mock resources to `rancher-desktop` namespace, ensure ContainerNamespace resources exist for volume namespaces.

### Important Issues

1. **`rancher-desktop` namespace overwrites its own ownerReference** — `pkg/controllers/mock/kube_namespace_reconciler.go:27-84` [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

```go
// SetupWithManager: both For and Watches target corev1.Namespace
For(&corev1.Namespace{}).
    Watches(&corev1.Namespace{},
        handler.EnqueueRequestForOwner(..., &corev1.Namespace{}, ...)).
    WithEventFilter(predicate.NewPredicateFuncs(func(object client.Object) bool {
        switch object.GetName() {
        case mockNamespaceName, apiNamespace:
            return true
        }
        // ...
    }))
```

The `For` handler enqueues the namespace name directly, while `Watches` with `EnqueueRequestForOwner` enqueues the owner. The predicate at line 75 admits both `rdd-mocks` and `rancher-desktop`. When `rancher-desktop` triggers an event, `For` enqueues `{Name: "rancher-desktop"}`. The reconciler at line 31 fetches `rancher-desktop`, builds an ownerReference from it (lines 41-47), and applies it back to `rancher-desktop` (lines 49-55) — a self-referential ownership.

This overwrites the correct `rdd-mocks` ownerReference set by the `{Name: "rdd-mocks"}` reconcile. The resulting oscillation flips ownership between `rdd-mocks` and `rancher-desktop` on each event. When `rancher-desktop` owns itself, deleting `rdd-mocks` orphans it. Codex confirmed this against a live control plane: `ownerReferences.name` alternated between `rancher-desktop` and `rdd-mocks` depending on event order.

All lines confirmed as commit 7ce6b31 via `git blame`.

Gemini rated this Critical; Claude and Codex rated it Important. Assessment: **Important.** The bug will manifest reliably, but its blast radius is limited to mock controller test infrastructure in an embedded single-user kube-apiserver. No production data loss or security impact.

Fix: Early-return in `Reconcile` for requests other than the intended trigger namespace:

```go
func (r *kubeNamespaceReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    if req.Name != mockNamespaceName {
        return ctrl.Result{}, nil
    }
    // ... rest of reconcile
}
```

The predicate must still admit `apiNamespace` so the `Watches` handler can detect changes to the owned namespace and enqueue `rdd-mocks`. The guard in `Reconcile` filters out spurious `For`-originated requests.

### Suggestions

1. **Missing `IsNotFound` handling causes retry churn on namespace deletion** — `pkg/controllers/mock/kube_namespace_reconciler.go:31-34` [Gemini 3.1 Pro]

```go
var rddNamespace corev1.Namespace
if err := r.Client.Get(ctx, req.NamespacedName, &rddNamespace); err != nil {
    log.Error(err, "Failed to get namespace", "namespace", req.NamespacedName)
    return ctrl.Result{}, err
}
```

If the watched namespace is deleted, `Get` returns `IsNotFound` and the returned error causes controller-runtime to requeue indefinitely with exponential backoff. This produces log noise during cleanup.

The same pattern exists in all other mock reconcilers (`container_reconciler.go:54`, `container_namespace_reconciler.go:48`, etc.), so this is a pre-existing gap carried forward into new code — not specific to this PR. The practical impact is low: exponential backoff limits CPU waste, and the namespace is deleted only during controller shutdown.

Fix: Handle `IsNotFound` explicitly:

```go
if err := r.Client.Get(ctx, req.NamespacedName, &rddNamespace); err != nil {
    if apierrors.IsNotFound(err) {
        return ctrl.Result{}, nil
    }
    log.Error(err, "Failed to get namespace", "namespace", req.NamespacedName)
    return ctrl.Result{}, err
}
```

2. **`getVolumeName` always returns a constant namespace** — `pkg/controllers/mock/volume_reconciler.go:42-44` [Claude Opus 4.6]

```go
func getVolumeName(volume mobyvolume.Volume) (namespace, name string) {
    return containerNamespace, volume.Name
}
```

The function ignores its argument for the namespace component — it always returns `"buildkit"`. The `containerNamespaceReconciler` iterates volumes via `getVolumeName` to collect namespaces (line 59-61), but the result is always the same value already seeded at line 58. The indirection is harmless but the name implies it extracts namespace from the volume data. A brief comment would prevent confusion.

Fix: Add a comment noting this is a placeholder for future namespace-per-volume logic.

### Design Observations

1. **`For` + `Watches` on the same type creates a footgun** (in-scope) [Claude Opus 4.6, Codex GPT 5.4]

The `kubeNamespaceReconciler` is the only mock reconciler where `For` and `Watches` target the same resource type (`corev1.Namespace`). Every other reconciler watches Namespace as the trigger and a distinct CRD type as the managed resource, avoiding cross-contamination between handlers. Using per-source predicates (`For(..., builder.WithPredicates(...))`) instead of a global `WithEventFilter` would eliminate the class of bugs where events from the managed object are misrouted through the `For` handler.

2. **A dedicated reconciler for a single static namespace adds complexity** (in-scope) [Codex GPT 5.4]

There is exactly one Kubernetes namespace to ensure: `rancher-desktop`. The existing `createNamespace` function in `mock_controller.go:119` already ensures `rdd-mocks` at startup. Ensuring `rancher-desktop` from the same `rdd-mocks`-driven reconcile path — or even from `createNamespace` itself — would remove the `Namespace -> Namespace` watch cycle entirely and eliminate the class of self-referential ownership bugs.

### Design Observations

**Splitting namespace creation into its own reconciler is sound** (in-scope)

The `kubeNamespaceReconciler` creates the `rancher-desktop` Kubernetes namespace as a prerequisite for all other mock resources. This establishes a clear dependency chain: `rdd-mocks` (trigger) -> `rancher-desktop` (API namespace) -> resources. Other reconcilers will fail and requeue until the namespace exists, which is standard controller behavior. The separation keeps each reconciler focused on a single concern.

**`For` + `Watches` on the same type requires care** (future)

The `kubeNamespaceReconciler` is the first mock reconciler that watches and manages the same resource type (`corev1.Namespace`) for both the trigger and the managed object. The other reconcilers avoid this because their managed types (Container, Image, Volume, ContainerNamespace) differ from the trigger type (Namespace). If future controllers need this pattern, consider using per-source predicates (`For(..., builder.WithPredicates(...))`) instead of a global `WithEventFilter`, which eliminates the cross-contamination between `For` and `Watches` handlers.

### Gemini 3.1 Pro

### Executive Summary

This PR correctly moves mock resources to the `rancher-desktop` namespace, ensures `ContainerNamespace` resources are created for mock volumes, and makes the `pid` field optional in the containers API. However, a logic flaw in the newly introduced `kubeNamespaceReconciler` causes the `rancher-desktop` namespace to overwrite its own `OwnerReference` to point to itself, severing its link to the `rdd-mocks` namespace and causing a resource leak upon mock deletion. The PR should be merged with fixes for this ownership corruption and missing error handling.

### Critical Issues

1. **`apiNamespace` overwrites its own owner reference, preventing garbage collection** — `pkg/controllers/mock/kube_namespace_reconciler.go:27-51` (critical, regression)

```go
		WithEventFilter(predicate.NewPredicateFuncs(func(object client.Object) bool {
			if _, ok := object.(*corev1.Namespace); ok {
				switch object.GetName() {
				case mockNamespaceName, apiNamespace:
					return true
				}
			}
			return false
		})).
```

Because `apiNamespace` ("rancher-desktop") is allowed by the event filter and the controller enqueues requests directly `For(&corev1.Namespace{})`, updates to `rancher-desktop` trigger a reconcile where `req.NamespacedName` is `"rancher-desktop"`. In `Reconcile`, the controller fetches the requested namespace, constructs an owner reference using its UID, and applies it to `"rancher-desktop"`. Using Server-Side Apply with the same `FieldManager` causes the list to be overwritten, replacing the ownership link to `rdd-mocks` with a link to itself. Consequently, when `rdd-mocks` is deleted, `rancher-desktop` will be orphaned and fail to garbage-collect.

**Fix:** In `Reconcile`, return early if the request is not for the controller's intended owner target:
```go
	if req.Name != mockNamespaceName {
		return ctrl.Result{}, nil
	}
```
*(Alternatively, remove `apiNamespace` from the `WithEventFilter` if `Watches` with `EnqueueRequestForOwner` is sufficient for tracking its deletion).*

### Important Issues

1. **Unhandled `IsNotFound` error causes infinite retry loops on deletion** — `pkg/controllers/mock/kube_namespace_reconciler.go:31-34` (important, gap)

```go
	var rddNamespace corev1.Namespace
	if err := r.Client.Get(ctx, req.NamespacedName, &rddNamespace); err != nil {
		log.Error(err, "Failed to get namespace", "namespace", req.NamespacedName)
		return ctrl.Result{}, err
	}
```

If the watched namespace (`rdd-mocks`) is deleted, `Get` will fail with a Kubernetes `IsNotFound` error. Because this error is returned directly, `controller-runtime` will continuously requeue the request with exponential backoff forever, polluting the logs and wasting CPU cycles.

**Fix:** Ignore the `IsNotFound` error to stop the retry loop, as Kubernetes owner-based garbage collection handles the cleanup of owned resources automatically upon deletion:
```go
	if err := r.Client.Get(ctx, req.NamespacedName, &rddNamespace); err != nil {
		if apierrors.IsNotFound(err) {
			return ctrl.Result{}, nil
		}
		log.Error(err, "Failed to get namespace", "namespace", req.NamespacedName)
		return ctrl.Result{}, err
	}
```
*(Note: This error-handling gap also exists in the other mock reconcilers such as `container_namespace_reconciler.go`, `container_reconciler.go`, etc., and should ideally be fixed for all of them.)*

### Design Observations

- **(in-scope) Future-proofing in `volume_reconciler`:** The loop introduced in `pkg/controllers/mock/container_namespace_reconciler.go` aggregates namespaces from all mock volumes. However, `getVolumeName` in `pkg/controllers/mock/volume_reconciler.go:42` currently hardcodes the return value to `"buildkit"`. While this structure works correctly and creates the expected `"buildkit"` `ContainerNamespace` today, Moby `Volume` objects do not natively expose a namespace concept. If test volumes are intended to exist in multiple namespaces in the future, `getVolumeName` will need a way to parse this data (e.g., via labels or name conventions) before this loop provides dynamic utility.

### Testing Assessment

1. **Untested scenario (High Risk):** Deletion lifecycle and Garbage Collection. The BATS tests correctly verify the creation of resources in the `rancher-desktop` namespace. However, there is no test verifying that deleting the `rdd-mocks` namespace correctly cascade-deletes the `rancher-desktop` namespace and its mock resources. The absence of a deletion test is what allowed the self-ownership regression to slip through.

### Documentation Assessment

The removal of `- pid` from the `required` list in `crd.yaml` and the addition of `+optional` in `pkg/apis/containers/v1alpha1/container_types.go` are self-documenting for CRD regeneration and align correctly. No further documentation updates are needed.
