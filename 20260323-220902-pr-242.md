## Executive Summary

This PR replaces WSL2-based BATS test execution with MSYS2 on Windows to eliminate flaky `binfmt_misc` interop failures, implements correct Windows process group isolation so hostagent lifecycle operations no longer kill the RDD service, adds WSL2 distro termination before unregistration to prevent `wslservice.exe` deadlocks, and pre-generates SSH keys to avoid MSYS2 path conversion issues. The implementation is thorough and well-documented, with defense-in-depth against WSL2 deadlocks and comprehensive SSH key crash recovery. The main finding is that the PID recycling guards added to the deletion path were not applied to the restart and orphan-recovery paths.

However, `handleWatchedState` (line 211, changed by commit `16c73c0`) and `handleUnwatchedState` (line 273, pre-existing) route `StatusBroken` instances through `stopInstanceForcibly` and `killOrphanedHostagent` without similar guards.

## Suggestions

1. **Unix `KillTree` returns nil for non-group-leader PIDs** — `process_unix.go:43-48` [Gemini 3.1 Pro] *(suggestion, gap)*

```go
func KillTree(_ context.Context, pid int) error {
    err := unix.Kill(-pid, unix.SIGKILL)
    if errors.Is(err, unix.ESRCH) {
        return nil
    }
    return err
}
```

`KillTree` sends SIGKILL to process group `-pid`. If the target is not a group leader (its PGID differs from its PID), the group doesn't exist and `Kill` returns `ESRCH`, which `KillTree` treats as "already dead." In `stopInstanceForcibly` (line 609), `KillTree` is called on both `DriverPID` and `HostAgentPID`. Since the driver inherits the hostagent's process group, `KillTree(driverPID)` would silently fail while `KillTree(hostagentPID)` kills the entire group including the driver.

Gemini rated this CRITICAL, but the practical risk is very low: on macOS (the only Unix platform), the VZ driver runs in-process so `DriverPID` is typically 0. Even with a separate driver (e.g., QEMU), the hostagent's `KillTree` call kills the entire group. The only scenario where this matters is `HostAgentPID == 0` and `DriverPID > 0` with a non-group-leader driver — an unlikely crash state.

Fix: Fall back to killing the individual process when the group doesn't exist:

```diff
 func KillTree(_ context.Context, pid int) error {
     err := unix.Kill(-pid, unix.SIGKILL)
     if errors.Is(err, unix.ESRCH) {
-        return nil
+        err = unix.Kill(pid, unix.SIGKILL)
+        if errors.Is(err, unix.ESRCH) {
+            return nil
+        }
     }
     return err
 }
```

## Design Observations

### Concerns

**Inconsistent PID trust model across lifecycle paths** *(in-scope)* [Codex GPT 5.4, Gemini 3.1 Pro] — The deletion path correctly distrusts stored PIDs for `StatusBroken` instances on Windows, but the restart and orphan-recovery paths trust them unconditionally. A unified approach — either validating process identity before killing (e.g., checking the executable name) or consistently skipping PID-based kills for `StatusBroken` — would eliminate the need to audit every new lifecycle path for the same guard.

### Strengths

- **Well-documented platform asymmetries** [Claude Opus 4.6] — Every divergence between Windows and Unix behavior (`KillTree` docstring, `stopInstanceForcibly`, `StopWithWait`, `shutdownHostagent`) has inline documentation explaining the difference and why it is acceptable. This is significantly above average for cross-platform code.

- **Thorough crash-state handling in `ensureSSHKeysAt`** [Claude Opus 4.6, Gemini 3.1 Pro] — Handles every combination of missing/corrupt/orphaned key state. Atomic rename-on-success prevents partial keys from blocking future attempts. Explicit temp file cleanup prevents `ssh-keygen` from hanging on overwrite prompts. Rollback of the public key when private key rename fails prevents partial state.

- **WSL2 deadlock defense-in-depth** [Claude Opus 4.6, Codex GPT 5.4] — Multiple layers protect against the `wslservice.exe` deadlock: `terminateWSL2Distro` before unregister in the controller, `taskkill /F /IM wslservice.exe` recovery in the BATS cleanup test, and timeouts on all `wsl.exe` calls.

- **`shutdownHostagent` consolidation** [Claude Opus 4.6] — Extracting the graceful-to-forced shutdown sequence into a single function eliminates duplicated shutdown logic that previously existed across `stopInstance`, `handleRestartNeeded`, and `shutdownAllHostagents`.

- **Fail-fast on dead processes** [Gemini 3.1 Pro] — In `shutdownHostagent`, if `process.Interrupt` returns an error (process already dead), it immediately falls through to forceful cleanup instead of waiting for the graceful timeout.

### Gemini 3.1 Pro

### Executive Summary

This PR migrates the Windows CI BATS tests from WSL2 to MSYS2 to bypass intermittent `binfmt_misc` interop failures. It introduces process group isolation to ensure RDD service stability during hostagent lifecycle events, and refactors Windows process signaling (`KillTree`) to terminate WSL2 distros and cleanly sweep orphaned port forwarders. The review found two critical regressions related to the newly introduced process tree termination logic: on Windows, blind usage of `taskkill` creates a severe risk of killing recycled PIDs, and on Unix, `KillTree` silently ignores single processes that aren't group leaders.

### Findings

#### Critical Issues

1. **`taskkill /F` introduces a critical risk of terminating unrelated processes due to PID recycling** — `pkg/util/process/process_windows.go:34-40`, `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:211` (critical, regression)

```go
// pkg/util/process/process_windows.go
func KillTree(ctx context.Context, pid int) error {
	err := exec.CommandContext(ctx, "taskkill", "/F", "/T", "/PID", strconv.Itoa(pid)).Run()
```

The PR replaces `limainstance.StopForcibly` with a custom `stopInstanceForcibly` that uses `process.KillTree` -> `taskkill /F /T /PID <pid>`. Lima's original `StopForcibly` on Windows uses `GenerateConsoleCtrlEvent`, which is safely scoped to the console group and benign if the PID was recycled to a random user process. However, `taskkill /F /T` operates system-wide. If Lima reads a stale PID file and the PID has been recycled to another user application (like an IDE or browser), `taskkill` will mercilessly terminate it and all its children.

The author acknowledges this risk in `handleDeletion` ("Broken instances may have stale PID files pointing to recycled processes"), skipping the kill for `StatusBroken`. However, this mitigation is incomplete:
- `handleWatchedState` (line 211) unconditionally calls `stopInstanceForcibly` for `StatusBroken` instances.
- `handleDeletion` still blindly kills recycled PIDs if BOTH the hostagent and driver PIDs happen to be recycled, resulting in `StatusRunning`.
- `killOrphanedHostagent` and `shutdownHostagent` also blindly pass stored PIDs to `stopInstanceForcibly`.

Furthermore, skipping the kill in `handleDeletion` for `StatusBroken` guarantees that a *genuinely* orphaned hostagent is never killed, causing a permanent resource leak.

**Fix:** Do not blindly execute `taskkill /PID` on PIDs derived from files. Instead, verify the process executable name before killing it, or at least skip the kill if the image name doesn't match expected patterns (`rdd.exe`, `wsl.exe`, etc.). 

```go
func isLimaProcess(ctx context.Context, pid int) bool {
	if runtime.GOOS != "windows" {
		return true 
	}
	out, err := exec.CommandContext(ctx, "tasklist", "/FI", fmt.Sprintf("PID eq %d", pid), "/NH").Output()
	if err != nil {
		return false
	}
	s := strings.ToLower(string(out))
	return strings.Contains(s, "rdd") || strings.Contains(s, "wsl") || strings.Contains(s, "qemu")
}
```
Apply this check in `stopInstanceForcibly` before invoking `KillTree(ctx, pid)`.

2. **`KillTree` on Unix silently fails to terminate processes that are not process group leaders, leaking VM drivers** — `pkg/util/process/process_unix.go:27-31` (critical, regression)

```go
func KillTree(_ context.Context, pid int) error {
	err := unix.Kill(-pid, unix.SIGKILL)
	if errors.Is(err, unix.ESRCH) {
		return nil
	}
	return err
}
```

`KillTree` sends `SIGKILL` to the process group (`-pid`). If the target process was not started as a process group leader (i.e., it does not have its own PGID matching its PID), the process group `-pid` does not exist. In this case, `unix.Kill` returns `ESRCH`. The code catches `ESRCH` and returns `nil`, silently assuming the process is already dead.

While the hostagent is explicitly started with `SetGroup`, the `DriverPID` (QEMU/VZ) inherits the hostagent's process group. If `stopInstanceForcibly` tries to kill the driver directly (e.g., if the hostagent PID file is missing so `HostAgentPID` is 0), `KillTree(ctx, inst.DriverPID)` will query a non-existent process group, receive `ESRCH`, and silently return `nil` without killing the driver. This leaks the QEMU/VZ process and leaves the VM running in the background.

**Fix:** If sending the signal to the process group returns `ESRCH`, fall back to sending it directly to the individual process.

```diff
 func KillTree(_ context.Context, pid int) error {
 	err := unix.Kill(-pid, unix.SIGKILL)
 	if errors.Is(err, unix.ESRCH) {
-		return nil
+		// Process group might not exist if target is not a group leader.
+		// Fall back to killing the process directly.
+		err = unix.Kill(pid, unix.SIGKILL)
+		if errors.Is(err, unix.ESRCH) {
+			return nil
+		}
 	}
 	return err
 }
```

#### Suggestions

3. **Explicitly remove destination files before renaming to ensure reliability on Windows** — `pkg/controllers/lima/limavm/controllers/ssh_keys.go:65-73` (suggestion, enhancement)

```go
	if err := os.Rename(tmpPath, privPath); err != nil {
		_ = os.Remove(pubPath)
		_ = os.Remove(tmpPath)
		return fmt.Errorf("failed to rename private key: %w", err)

```

On Windows, `os.Rename` fails if the destination file already exists.
