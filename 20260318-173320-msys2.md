## Consolidated Review

### Executive Summary

This branch enables Lima BATS tests on Windows (MSYS2/WSL2) by replacing Lima's upstream `StopForcibly` -- whose `GenerateConsoleCtrlEvent(CTRL_BREAK)` kills the entire console group, including the RDD service -- with a custom `stopInstanceForcibly` that uses per-process `TerminateProcess`. The BATS tests gain cross-platform process helpers, WSL2-specific templates, and distro cleanup in teardown. The implementation solves the core Windows problem. The principal regression is that `killOrphanedHostagent` now force-kills on all platforms, removing the graceful shutdown path that Unix previously had.

**Structure:** 3 commits, 9 files, +220/-79 lines. Go changes in `limavm_lifecycle.go` (shutdown paths), `hostagent_watcher.go` (signalHostagent refactor), and `process_{unix,windows}.go` (new `Interrupt`/`Kill` helpers). BATS changes in `os.bash` (cross-platform process helpers), `utils.bash`, `limavm-instance.bats`, and `limavm-running.bats`.

### Critical Issues

None.

### Important Issues

## Design Observations

1. **Platform-specific force-stop strategy** *(in-scope)* [Codex GPT 5.4, Gemini 3.1 Pro]

The Windows workaround replaces Lima's stop helpers on all platforms. A cleaner design isolates the Windows-specific `TerminateProcess` logic behind a platform build tag while preserving Lima's upstream behavior on Unix. This eliminates a class of drift bugs (like the missing disk unlock) and keeps RDD aligned with Lima's evolving cleanup logic on platforms where Lima works correctly. ```go

### Gemini 3.1 Pro

### Executive Summary

This PR successfully adapts process signaling and lifecycle management for Windows and WSL2, fixing BATS tests and introducing a `CREATE_NEW_PROCESS_GROUP` strategy to bypass Lima's destructive `SysKill` behavior on Windows. However, it introduces critical regressions in VM shutdown logic: orphaned hostagents are now forcefully hard-killed upon controller restart (risking data corruption), and failing to gracefully signal a hostagent leaves the VM driver (WSL2/QEMU) secretly running in the background.

### Findings

#### Critical Issues

1. **Orphaned hostagents are forcefully killed without graceful shutdown** -- `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:504-507` `(critical, regression)`

```go
// killOrphanedHostagent terminates an orphaned hostagent (one running without a
// watcher, typically after a controller restart).
func (r *LimaVMReconciler) killOrphanedHostagent(_ context.Context, inst *limatype.Instance) error {
	stopInstanceForcibly(inst)
	return nil
}
```

The PR replaces `limainstance.StopGracefully` with an immediate `stopInstanceForcibly(inst)` for orphaned hostagents. While removing Lima's `StopGracefully` is necessary to prevent a crash loop on Windows (as its `SysKill` implementation targets the entire console group), completely dropping graceful shutdown is unsafe. It causes the controller to forcefully hard-kill running VMs (both the driver and hostagent) whenever the RDD daemon restarts, denying the guest OS the opportunity to sync filesystems and cleanly shut down. This poses a high risk of user data corruption.

**Fix:** Reimplement a safe graceful shutdown within `killOrphanedHostagent` using the new `process.Interrupt(inst.HostAgentPID)` to signal the hostagent's process group. Wait for the process to exit (or poll `store.Inspect` until it reports `Stopped`), and only fall back to `stopInstanceForcibly` if the timeout expires.

2. **Failing to signal the hostagent leaves the VM driver orphaned and leaks resources** -- `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:466-473` `(critical, regression)`

```go
	} else {
		// Signal delivery failed (e.g. os.Interrupt is not supported on Windows).
		// Kill the hostagent process directly via its process handle rather than
		// via StopForcibly (which kills by PID and cleans up files concurrently).
		logger.Info("Could not signal hostagent, killing process directly")
		r.killHostagent(limaVM.Name)
		r.waitForProcessExit(ctx, limaVM.Name)
	}
```

In `stopInstance` (and identically around line 295 in `handleRestartNeeded`), if `r.signalHostagent` fails or returns false because the watcher state is lost, the fallback branch calls `r.killHostagent(limaVM.Name)`. `killHostagent` *only* terminates the hostagent process; it does not terminate the VM driver (QEMU or WSL2 keepalive) or clean up tmp files. Because `stopInstanceForcibly(inst)` is intentionally skipped in this branch, the driver process is left orphaned and running indefinitely. The instance silently enters a `Broken` state (hostagent dead, driver alive), which bypasses the `inst.Status == limatype.StatusRunning` error check, falsely marking the CR as `Stopped` while the VM continues to consume system resources.

**Fix:** In the `else` block, after killing the hostagent process (or if the state is missing), you must call `stopInstanceForcibly(inst)` to guarantee the driver is terminated and the instance directory is fully cleaned up.

#### Important Issues

1. **Unbounded execution of `wsl.exe` can permanently hang the reconciler** -- `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:531-533` `(important, gap)`

```go
	if inst.VMType == limatype.WSL2 {
		distroName := "lima-" + inst.Name
		// Best-effort: ignore errors (distro may already be stopped).
		_ = exec.Command("wsl.exe", "--terminate", distroName).Run()
	}
```

The `wsl.exe` command is executed synchronously without a timeout. If the WSL subsystem enters a degraded or unresponsive state (a known failure mode for WSL2), `wsl.exe` can hang indefinitely. Because this is called directly within the reconcile loop, a hang will permanently block the worker goroutine, potentially starving the controller-runtime's worker pool and halting all VM lifecycle operations.

**Fix:** Use `context.WithTimeout` and `exec.CommandContext` to enforce a strict timeout (e.g., 15-30 seconds) on the `wsl.exe` execution.

2. **`filepath.Glob` usage breaks tmp file cleanup if paths contain metacharacters** -- `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:536-540` `(important, regression)`

```go
	for _, suffix := range filenames.TmpFileSuffixes {
		matches, _ := filepath.Glob(filepath.Join(inst.Dir, "*"+suffix))
		for _, m := range matches {
			_ = os.Remove(m)
		}
	}
```

`filepath.Glob` treats characters like `[` and `]` as pattern metacharacters. If `inst.Dir` (which derives from `RDD_LIMA_HOME` or the user's profile path) contains these characters, the pattern will fail to match silently. The `.pid` and `.sock` files will be left behind in the instance directory, preventing the VM from successfully restarting until manually deleted.

**Fix:** Mirror Lima's original `StopForcibly` logic: use `os.ReadDir(inst.Dir)` to list files and `strings.HasSuffix(f.Name(), suffix)` to match them safely regardless of the directory path's contents.

#### Suggestions

1. **Unintended PID targeting risk on Windows** -- `pkg/controllers/lima/limavm/controllers/limavm_lifecycle.go:521-527` `(suggestion, gap)`
On Windows, `os.FindProcess(pid)` succeeds even if the process has terminated, returning a handle to whatever process currently holds that PID. If the original driver or hostagent crashed and the OS recycled the PID quickly, `p.Kill()` will forcefully terminate an unrelated application. While this risk is inherent to relying on PIDs recorded in files (and existed in Lima's `osutil.SysKill`), it is worth noting. Process handles (like `state.cmd.Process`) should be preferred over PIDs when available.

### Design Observations

**Duplicate cleanup logic vs upstreaming** `(future)`
The PR hand-rolls a simplified version of Lima's `StopForcibly` logic into `stopInstanceForcibly` because Lima's `osutil.SysKill` crashes the RDD service on Windows. Rather than maintaining a parallel implementation (which currently omits Lima's disk-unlocking logic and risks diverging over time), consider contributing the `CREATE_NEW_PROCESS_GROUP` strategy upstream to Lima. If `osutil.SysKill(..., osutil.SigInt)` worked safely on Windows, RDD could rely natively on Lima's `StopGracefully` and `StopForcibly` APIs again.

### Testing Assessment

The BATS tests were successfully updated to pass on Windows by replacing raw Unix commands with `assert_process_alive` and fixing WSL path conversions. However, test coverage has the following gaps (ranked by risk):

1. **Orphaned hostagent shutdown safety:** The test "restart service and verify orphaned hostagent recovery" verifies that the old PID is dead, but it does not verify *how* it died. Because of the regression in `killOrphanedHostagent`, this test passes even though the VM is now force-killed. A test should verify that a graceful shutdown occurred (e.g., by checking if a shutdown trap script fired inside the guest OS).
2. **Driver termination verification:** Tests do not verify that the driver (QEMU/WSL) is actually terminated when the hostagent crashes or fails to stop, which would have caught the leaked WSL distro bug in the `stopInstance` fallback block.
3. **WSL timeout resilience:** There is no test simulating a hung `wsl.exe` command to ensure the controller doesn't block forever.

### Documentation Assessment

The new `stopInstanceForcibly` function lacks comments explaining why it intentionally omits the disk-unlocking steps found in Lima's original `StopForcibly`. If RDD doesn't use `AdditionalDisks`, a comment should state this assumption so future maintainers don't introduce disks and subsequently encounter locked-disk errors during forceful stops.
