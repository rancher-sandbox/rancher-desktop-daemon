## Consolidated Review

### Executive Summary

This branch fixes a critical Windows bug where Lima's `StopForcibly` uses `GenerateConsoleCtrlEvent(CTRL_BREAK)` to kill hostagents, which targets the entire console group and kills the RDD service along with the hostagent. The fix introduces `CREATE_NEW_PROCESS_GROUP` for hostagent processes and a custom `stopInstanceForcibly` that uses `os.Process.Kill` (`TerminateProcess`) targeted at individual processes. The branch also adds WSL2 distro termination, SSH key pre-generation for Windows, and migrates BATS CI from WSL2 to MSYS2.

The custom `stopInstanceForcibly` replaces upstream Lima's `StopForcibly` but drops two behaviors: child process cleanup on Windows and the safer `os.ReadDir` approach for temp file removal. The deletion path now removes the cleanup finalizer even when backend cleanup fails, a deliberate trade-off for WSL2 resilience that warrants discussion.

### Critical Issues

None.

### Important Issues

1. **`stopInstanceForcibly` leaves orphaned child processes on Windows** — `limavm_lifecycle.go:539-545` [Gemini 3.1 Pro]

```go
for _, pid := range []int{inst.DriverPID, inst.HostAgentPID} {
    if pid > 0 {
        if p, err := os.FindProcess(pid); err == nil {
            _ = p.Kill()
        }
    }
}
```

`os.Process.Kill()` calls `TerminateProcess` on Windows, which terminates only the specified process. Lima's hostagent spawns `ssh.exe` for port forwarding; killing the hostagent without killing its children leaves those `ssh.exe` processes holding ports. The next hostagent startup fails with "address already in use." On WSL2, `wsl.exe --terminate` (line 553) cleans up in-distro processes but not host-side port forwarders. This regression was introduced in `be9c281`. (important, regression)

Fix: Use `taskkill /F /T /PID <pid>` on Windows to terminate the process tree, or enumerate and kill child processes via the Windows job object API.

2. **`stopInstanceForcibly` uses `filepath.Glob`, which fails on paths with bracket characters** — `limavm_lifecycle.go:557-561` [Gemini 3.1 Pro]

```go
for _, suffix := range filenames.TmpFileSuffixes {
    matches, _ := filepath.Glob(filepath.Join(inst.Dir, "*"+suffix))
    for _, m := range matches {
        _ = os.Remove(m)
    }
}
```

Lima's upstream `StopForcibly` (at `stop.go:143-162`) uses `os.ReadDir` + `strings.HasSuffix`. The new code uses `filepath.Glob`, which treats `[` and `]` as glob meta-characters. If the user's home directory path contains brackets (e.g., `C:\Users\name[1]`), `filepath.Glob` returns `ErrBadPattern` (silently discarded), leaving `.pid` and `.sock` files behind. The next hostagent fails to start. Introduced in `1f0db09`. (important, regression)

Fix: Match upstream's approach:
```go
fi, err := os.ReadDir(inst.Dir)
if err == nil {
    for _, f := range fi {
        for _, suffix := range filenames.TmpFileSuffixes {
            if strings.HasSuffix(f.Name(), suffix) {
                _ = os.Remove(filepath.Join(inst.Dir, f.Name()))
            }
        }
    }
}
```

3. **Deletion removes cleanup finalizer after backend delete failure** — `limavm_lifecycle.go:53-63` [Codex GPT 5.4]

```go
deleteCtx, deleteCancel := context.WithTimeout(ctx, 60*time.Second)
err = limainstance.Delete(deleteCtx, existingInst, true)
deleteCancel()
if err != nil {
    logger.Error(err, "Failed to delete Lima instance (proceeding with finalizer removal)")
}
```

When `limainstance.Delete` fails (line 56), the reconciler logs the error and continues to remove the cleanup finalizer at line 80. Once the finalizer is gone, no retry mechanism exists; a failed `wsl.exe --unregister` strands the WSL2 distro permanently. The comment at lines 59-62 documents this as a deliberate trade-off to prevent the K8s resource from getting stuck, and manual cleanup via `wsl --unregister` is possible. However, the trade-off means a transient WSL2 glitch (timeout rather than permanent failure) causes permanent resource leakage with no automated recovery. Introduced in `1f0db09`. (important, regression)

Fix: Distinguish transient from permanent failures. On timeout, requeue with backoff (perhaps with a retry counter) instead of removing the finalizer immediately. Remove the finalizer unconditionally only after N retries or on a non-retryable error (e.g., "instance not found").

4. **`shutdownAllHostagents` hard-kill fallback skips WSL2 distro termination and temp file cleanup** — `limavm_controller.go:508-512` [Claude Opus 4.6]

```go
case <-time.After(gracefulShutdownTimeout):
    if state.cmd != nil && state.cmd.Process != nil {
        _ = state.cmd.Process.Kill()
    }
    <-state.procExited
```

Every other forced-shutdown path (`stopInstance` at line 479, `handleRestartNeeded` at line 301, `handleDeletion` at line 50, `killOrphanedHostagent` at line 525) calls `stopInstanceForcibly`, which terminates the WSL2 distro and cleans up `.pid`/`.sock`/`.tmp` files. This fallback does `Process.Kill()` alone, leaving WSL2 distros running after `rdd svc stop` if a hostagent hangs. Pre-existing gap, but amplified by this PR establishing `stopInstanceForcibly` as the canonical cleanup path everywhere else. (important, gap)

Fix: Pass a `context.Context` to `shutdownAllHostagents` and call `store.Inspect` + `stopInstanceForcibly` after the kill fallback fires. This is a structural change suitable for a follow-up.

### Suggestions

1. **`stopInstanceForcibly` silently discards all errors** — `limavm_lifecycle.go:538-563` [Claude Opus 4.6]

```go
func stopInstanceForcibly(ctx context.Context, inst *limatype.Instance) {
    // every _ = p.Kill(), _ = exec.CommandContext(...).Run(), _ = os.Remove(m)
```

Every error is discarded: process kill (line 542), WSL2 terminate (line 553), and file removal (line 560). A failed `wsl.exe --terminate` or process kill produces no log entry. The function is a standalone helper with no logger. (suggestion, gap)

Fix: Accept a `logr.Logger` parameter and log failed cleanup steps at V(1). Alternatively, make it a method on `LimaVMReconciler` to access the logger from context.

2. **`SetGroup` overwrites `SysProcAttr` instead of merging** — `process_windows.go:20-22` [Gemini 3.1 Pro]

```go
func SetGroup(cmd *exec.Cmd) {
    cmd.SysProcAttr = &windows.SysProcAttr{CreationFlags: windows.CREATE_NEW_PROCESS_GROUP}
}
```

Both the Windows and Unix implementations overwrite `cmd.SysProcAttr` entirely. If a future caller sets other process attributes before calling `SetGroup`, those settings are silently lost. Currently, the only call site (`startInstance` at line 423) creates a fresh `exec.Cmd`, so the overwrite is safe. (suggestion, gap)

Fix: Initialize `SysProcAttr` only if nil, then merge flags:
```go
if cmd.SysProcAttr == nil {
    cmd.SysProcAttr = &windows.SysProcAttr{}
}
cmd.SysProcAttr.CreationFlags |= windows.CREATE_NEW_PROCESS_GROUP
```

3. **Duplicated shutdown pattern in `stopInstance` and `handleRestartNeeded`** — `limavm_lifecycle.go:289-316` vs `limavm_lifecycle.go:472-490` [Claude Opus 4.6]

Both functions implement the same signal-wait-forcestop pattern. The two implementations differ slightly (`handleRestartNeeded` re-inspects via `store.Inspect` before `stopInstanceForcibly`; `stopInstance` reuses its `inst` parameter). As both grow, they risk diverging. (suggestion, gap)

Fix: Extract a `shutdownHostagent(ctx, name, inst)` helper that encapsulates the signal-or-kill-then-cleanup pattern.

4. **`killOrphanedHostagent` dropped graceful shutdown on Unix** — `limavm_lifecycle.go:522-527` [Claude Opus 4.6]

```go
func (r *LimaVMReconciler) killOrphanedHostagent(ctx context.Context, inst *limatype.Instance) error {
    stopInstanceForcibly(ctx, inst)
    return nil
}
```

The previous implementation tried `StopGracefully` (SIGTERM + 30s wait) before falling back to `StopForcibly`. The new code goes straight to hard kill. On Windows, the old graceful path was broken (`os.Process.Signal(os.Interrupt)` always fails), so this is a net improvement there. On Unix, orphaned hostagents lose their chance to shut down the VM driver gracefully. For post-crash recovery, speed matters more than cleanliness, so the trade-off is reasonable. (suggestion, regression)

5. **SSH key race between concurrent controller startups** — `ssh_keys_windows.go:25-42`, `limavm_controller.go:452-453` [Codex GPT 5.4]

```go
if _, err := os.Stat(privPath); err == nil {
    return nil
}
// ...
cmd := exec.CommandContext(ctx, "ssh-keygen", "-t", "ed25519", ...)
```

`ensureSSHKeys` has a check-then-act race: two controllers starting simultaneously against the same Lima config dir can both observe "key missing" and race `ssh-keygen`. Lima upstream uses `lockutil.WithDirLock` to serialize this. In practice, the window is narrow and `ssh-keygen` prompts to overwrite (failing without a tty), so the second instance would return an error and prevent startup. (suggestion, regression)

Fix: Wrap the `ssh-keygen` call in Lima's `lockutil.WithDirLock(configDir, ...)`.

## Design Observations

### Concerns

1. **`stopInstanceForcibly` forks upstream cleanup semantics instead of overriding just the kill primitive** — [Codex GPT 5.4, Claude Opus 4.6] (in-scope)

The new `stopInstanceForcibly` replaces the entire `StopForcibly` function rather than overriding just the Windows kill call. This already caused the `filepath.Glob` regression and the orphaned-children issue, and will drift further as Lima evolves (e.g., the additional-disk unlock at `stop.go:119-128`, which RDD doesn't currently use but might in the future). A better structure: keep one local helper for "Windows-safe kill PID" and call upstream `StopForcibly` for everything else, or copy upstream's cleanup logic faithfully when the kill primitive must differ.

### Strengths

1. **`CREATE_NEW_PROCESS_GROUP` + targeted `GenerateConsoleCtrlEvent` is the right approach** — [Claude Opus 4.6] (in-scope) — The fix isolates the hostagent into its own process group and targets `CTRL_BREAK_EVENT` at that group specifically. Correct and clean.

2. **SSH key pre-generation is a clean Lima workaround** — [Claude Opus 4.6] (in-scope) — `ensureSSHKeys` runs at controller startup, generates the keypair with a native Windows path, and is idempotent. Avoids patching Lima while keeping the fix local to RDD.

3. **BATS helpers provide good cross-platform abstraction** — [Claude Opus 4.6] (in-scope) — `assert_process_alive` and `process_kill` correctly handle the MSYS2-to-Win32 process boundary. The `vm_template.bash` module cleanly separates platform-specific template selection.

**Executive Summary**

This series moves Windows BATS coverage from WSL to MSYS2, reworks Lima hostagent signaling on Windows, pre-generates Lima SSH keys on Windows, and changes forcible stop/delete handling for WSL2-backed instances. The Windows process work is directionally right, but I found two important regressions in the new force-stop/delete path: deletion now drops the cleanup finalizer even when backend cleanup fails, and the replacement `stopInstanceForcibly()` silently stopped unlocking Lima additional disks.

**Important Issues**

1. **Deletion now discards the only cleanup retry path after a failed backend delete** — `limavm_lifecycle.go:52-63` (important, regression)

```go
deleteCtx, deleteCancel := context.WithTimeout(ctx, 60*time.Second)
err = limainstance.Delete(deleteCtx, existingInst, true)
deleteCancel()
if err != nil {
	logger.Error(err, "Failed to delete Lima instance (proceeding with finalizer removal)")
}
```

The new branch at line 56 deliberately continues after `limainstance.Delete(...)` fails at line 56, and the reconcile then proceeds to remove the cleanup finalizer via `base.RemoveCleanupFinalizer()` in `pkg/controllers/base/finalizer.go:48-55`. That is a real behavior change: the base finalizer is documented at `pkg/controllers/base/finalizer.go:25-27` as the mechanism that ensures Lima disk/file cleanup runs before the CR is deleted. Once line 63 logs and continues, there is no owner object left to retry cleanup, so a timed-out or failed `Delete` can strand the on-disk instance and WSL distro permanently. `git blame` attributes lines 53-63 to `1f0db09`, so this regression is introduced by this series.

Fix: if `limainstance.Delete()` fails at line 56, keep the cleanup finalizer and return an error or `RequeueAfter` so deletion can be retried. If you need an escape hatch for degraded WSL, gate "force-remove finalizer anyway" behind an explicit user opt-in or a separate admin cleanup flow.

2. **The custom force-stop helper dropped Lima's additional-disk unlock step** — `limavm_lifecycle.go:538-562` (important, regression)

```go
func stopInstanceForcibly(ctx context.Context, inst *limatype.Instance) {
	for _, pid := range []int{inst.DriverPID, inst.HostAgentPID} {
		if pid > 0 {
			if p, err := os.FindProcess(pid); err == nil {
				_ = p.Kill()
			}
		}
	}
	// ...
}
```

The replacement `stopInstanceForcibly()` at lines 538-562 is now used from deletion, restart recovery, stop fallback, and orphan cleanup, but it only kills processes and removes tmp files. The simplest fix is to copy the additional-disk unlock loop from upstream `StopForcibly()` into this helper, or refactor so only the signal/kill primitive is overridden while upstream cleanup logic stays intact.


`SetupWithManager()` now calls `ensureSSHKeys()` on every controller startup at lines 452-453, but `ensureSSHKeys()` at lines 30-42 is just a `Stat` plus `ssh-keygen`. Upstream Lima's `sshutil.DefaultPubKeys()` uses `lockutil.WithDirLock(configDir, ...)` at `pkg/sshutil/sshutil.go:109-125` specifically to serialize first-time key generation. This repo explicitly supports multiple RDD instances via `RDD_INSTANCE` in `pkg/instance/instance.go:15-37`, so two Windows controllers starting against the same Lima config dir can both observe "key missing" and race one another. `git blame` points both the setup call and helper implementation to `625737d`.

Fix: wrap the new Windows-native `ssh-keygen` call in Lima's existing `lockutil.WithDirLock(configDir, ...)`, mirroring upstream behavior while keeping the native-path workaround.

**Design Observations**

1. (in-scope) The new `stopInstanceForcibly()` forks upstream Lima cleanup semantics instead of just swapping out the Windows kill primitive. That already caused the missed additional-disk unlock, and it will keep drifting as Lima evolves. A better structure is to keep one local helper for "Windows-safe kill PID/process-group" and reuse the rest of Lima's force-stop flow unchanged; that removes a whole class of parity bugs.

**Testing Assessment**

1. No test covers force-stopping or deleting a Lima VM that uses `additionalDisks`, so the disk-unlock regression above would not be caught.
2. No test exercises the delete-timeout/delete-failure path on WSL2, so the new "log and remove finalizer anyway" behavior is unverified.
3. No test covers concurrent first-start of multiple Windows RDD instances sharing one Lima config dir, so the new SSH-key generation path is untested under the race that upstream Lima locks against.

**Documentation Assessment**

The user-facing behavior change around failed WSL2 deletion is significant enough that it should not live only in comments. If the intent is really to allow CR deletion while backend cleanup fails, that needs explicit operator documentation and a supported manual cleanup procedure; otherwise the code should keep the existing retry semantics.

### Gemini 3.1 Pro

#### Executive Summary

This PR successfully migrates the Lima BATS integration tests to run via MSYS2 on Windows and addresses several Windows-specific lifecycle issues, including pre-generating SSH keys to sidestep MSYS2 path translation bugs and replacing Lima's forceful shutdown logic to avoid inadvertently killing the parent service. The primary findings revolve around edge cases introduced by the new lifecycle process handling: an API design gap that blindly overwrites `SysProcAttr`, a regression in how temporary files are cleaned up during forceful termination, and a risk of leaving orphaned `ssh.exe` port forwarders running in the background on Windows.

#### Critical Issues
None.

#### Important Issues

1. **`SetGroup` blindly overwrites `SysProcAttr` instead of modifying it** — `process_windows.go:20-22`

```go
func SetGroup(cmd *exec.Cmd) {
	cmd.SysProcAttr = &windows.SysProcAttr{CreationFlags: windows.CREATE_NEW_PROCESS_GROUP}
}
```

The new cross-platform `SetGroup` function overwrites `cmd.SysProcAttr` entirely. If the caller had already configured other process attributes (e.g., `HideWindow` on Windows, or credentials on Unix), those settings would be silently discarded. This creates a brittle API that assumes it is the sole author of `SysProcAttr` properties.

Fix: Modify the struct if it already exists, or initialize it if nil. Use bitwise operations where appropriate:
```go
func SetGroup(cmd *exec.Cmd) {
	if cmd.SysProcAttr == nil {
		cmd.SysProcAttr = &windows.SysProcAttr{}
	}
	cmd.SysProcAttr.CreationFlags |= windows.CREATE_NEW_PROCESS_GROUP
}
```
*(Apply similarly for `process_unix.go` lines 17-19 using `cmd.SysProcAttr.Setpgid = true`)* (important, gap)

*Consolidation note: The only call site creates a fresh `exec.Cmd`, so the overwrite is safe today. Downgraded to suggestion in the consolidated review.*

2. **`stopInstanceForcibly` uses `filepath.Glob` which breaks on path meta-characters** — `limavm_lifecycle.go:550-555`

```go
	for _, suffix := range filenames.TmpFileSuffixes {
		matches, _ := filepath.Glob(filepath.Join(inst.Dir, "*"+suffix))
		for _, m := range matches {
			_ = os.Remove(m)
		}
	}
```

The original `limainstance.StopForcibly` logic explicitly used `os.ReadDir` and `strings.HasSuffix`. The new implementation switches to `filepath.Glob`. If `inst.Dir` (often located in a user's home directory) contains glob meta-characters like `[` or `]`, `filepath.Glob` will fail with `ErrBadPattern` or silently return no matches. This will leave `.pid` and `.sock` files behind, which prevents the hostagent from successfully starting on the next attempt.

Fix: Revert to the safer directory traversal approach originally used by Lima:
```go
	fi, err := os.ReadDir(inst.Dir)
	if err == nil {
		for _, f := range fi {
			for _, suffix := range filenames.TmpFileSuffixes {
				if strings.HasSuffix(f.Name(), suffix) {
					_ = os.Remove(filepath.Join(inst.Dir, f.Name()))
				}
			}
		}
	}
```
(important, regression)

3. **`stopInstanceForcibly` risks leaving orphaned child processes on Windows** — `limavm_lifecycle.go:535-541`

```go
	for _, pid := range []int{inst.DriverPID, inst.HostAgentPID} {
		if pid > 0 {
			if p, err := os.FindProcess(pid); err == nil {
				_ = p.Kill()
			}
		}
	}
```

The PR replaces Lima's `GenerateConsoleCtrlEvent(CTRL_BREAK_EVENT)` with Go's `p.Kill()`, which invokes `TerminateProcess` under the hood on Windows. While this successfully avoids killing the parent RDD service, `TerminateProcess` does not terminate child processes. Since Lima's hostagent spawns `ssh.exe` for port forwarding on Windows, forcefully killing the hostagent leaves these `ssh.exe` processes orphaned in the background. The surviving orphans will continue to hold locks on forwarded ports, triggering "address already in use" errors and blocking subsequent VM startups.

Fix: For forceful termination on Windows, consider using `taskkill /F /T /PID <pid>` to cleanly kill the process tree. Alternatively, document this limitation and explicitly clean up known child processes if they can be tracked. (important, regression)

#### Suggestions

1. **Improve PID matching precision in `assert_process_alive`** — `os.bash:103-104`

```bash
        MSYS_NO_PATHCONV=1 tasklist.exe /FI "PID eq ${pid}" /NH 2>/dev/null | grep -q "${pid}"
```

While `tasklist` filters by the exact PID, using `grep -q "${pid}"` could technically match the PID substring inside the process name (e.g., matching "12" against `test12.exe`). Since `tasklist` outputs `INFO: No tasks are running...` when no match is found, this is functionally safe in practice, but adding the word boundary flag (`-w`) or extended regex (`-E "\b${pid}\b"`) prevents any theoretical edge cases. (suggestion, enhancement)

#### Design Observations

**Graceful shutdown synchronization on Windows (in-scope)**
In `limavm-running.bats` (lines 332-337), you skip waiting for the hostagent PID to die on Windows, citing child process handles keeping the PID visible in the process table. In reality, Windows `tasklist` queries do not show zombie processes; if `tasklist` shows the PID, the process has not terminated. This implies the hostagent is genuinely blocking during its graceful shutdown phase (likely waiting on its `wsl.exe` driver or `ssh.exe` children that might ignore the `CTRL_BREAK_EVENT`), eventually forcing the service to hard-kill it during shutdown. Tracing why the hostagent blocks on `CTRL_BREAK_EVENT` on Windows could eliminate the need for this test workaround and improve overall shutdown reliability.
