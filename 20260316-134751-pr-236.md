## Consolidated Review

### Executive Summary

This PR adds the App controller's core reconciliation loop: creating a LimaVM from an embedded template, propagating `spec.running`, mirroring status conditions, and deleting owned resources via a finalizer. The design is well-structured, with correct use of existing patterns for finalizer management and owner references. However, a critical deadlock prevents App deletion from ever completing: the App controller sets `OwnedFinalizerName` on the LimaVM at creation but never removes it during teardown. The embedded template also hard-codes amd64, which breaks Apple Silicon.

---

### Critical Issues

1. **LimaVM stuck in Terminating: `/owned` finalizer never removed** — `pkg/controllers/app/app/controllers/app_controller.go:75-84` [Gemini 3.1 Pro]

```go
case !base.IsBeingDeleted(limaVM):
    if err := r.Delete(ctx, limaVM); err != nil && !apierrors.IsNotFound(err) {
        return ctrl.Result{}, fmt.Errorf("failed to delete LimaVM: %w", err)
    }
    log.Info("Requested LimaVM deletion, waiting for teardown")
    return ctrl.Result{RequeueAfter: requeueAfterDeletion}, nil
default:
    // LimaVM deletion is in progress; keep requeueing until it is gone.
    log.Info("Waiting for LimaVM deletion to complete")
    return ctrl.Result{RequeueAfter: requeueAfterDeletion}, nil
```

The App controller creates the LimaVM with `OwnedFinalizerName` at line 132. During App deletion, the App controller calls `r.Delete(ctx, limaVM)` (line 76) and polls until the LimaVM disappears. But it never strips `OwnedFinalizerName` from the LimaVM. The LimaVM controller's `handleDeletion` (`limavm_lifecycle.go:32-78`) removes only `CleanupFinalizerName` (line 71) — its `DeleteOwnedResources` call (line 65) strips `/owned` from resources owned *by* the LimaVM (ConfigMaps), not from the LimaVM itself. With `/owned` still present, the LimaVM remains in Terminating permanently, and the App controller polls forever.

The BATS test at `app.bats:177-179` (`rdd ctl wait --for=delete app/"${APP_NAME}" --timeout=90s`) will hang and time out.

Fix: Either call `base.RemoveOwnedFinalizer(ctx, r.Client, limaVM)` before issuing the delete, or replace the manual find-and-delete with `base.DeleteOwnedResources(ctx, r.Client, &app, r.Manager)`, which strips `/owned` from all owned resources and deletes them in one pass. The second option also handles the orphan race in finding #3.

### Important Issues

1. **Embedded template hard-codes amd64 — arm64 hosts will fail** — `pkg/controllers/app/app/lima.yaml:3` [Claude Opus 4.6, Codex GPT 5.4, Gemini 3.1 Pro]

```yaml
images:
- location: https://github.com/rancher-sandbox/rancher-desktop-opensuse/releases/download/v0.1.0/distro.v0.1.0.amd64.qcow2
```

The template points to `distro.v0.1.0.amd64.qcow2` unconditionally. On arm64 hosts (Apple Silicon), Lima will fail to match a suitable image, and `limainstance.Prepare()` will fail. Rosetta is also disabled (`rosetta: enabled: false`). Issue #227 explicitly requires "Select correct arch (amd64/arm64) and image format (qcow2/tar) from host."

The template is marked temporary (`controller.go:18`), so this may be intentional as a first pass. But the BATS test cannot pass on arm64 as-is.

Fix: Generate the template dynamically (e.g., substitute the URL based on `runtime.GOARCH`), or add both architecture entries to the image list (Lima selects the matching one), or gate the test with a skip guard.

2. **Orphaned `rd-template` ConfigMap in pre-adoption deletion race** — `pkg/controllers/app/app/controllers/app_controller.go:62-84` [Codex GPT 5.4]

```go
case apierrors.IsNotFound(err):
    // LimaVM is gone. Clean up the input ConfigMap if it still exists
    inputCM := &corev1.ConfigMap{}
    if cmErr := r.Get(ctx, client.ObjectKey{Name: inputConfigMapName, Namespace: namespace}, inputCM); cmErr == nil {
        if cmErr := r.Delete(ctx, inputCM); cmErr != nil && !apierrors.IsNotFound(cmErr) {
            return ctrl.Result{}, fmt.Errorf("failed to delete input ConfigMap during cleanup: %w", cmErr)
        }
    }
```

The LimaVM webhook creates `rd-template` with `OwnedFinalizerName` but without an ownerReference (line 210-212 of `limavm/controller.go`). The reconciler adds the ownerReference in a subsequent reconcile (line 232-242). If the App is deleted during this window, the App controller's cleanup (lines 62-71) only looks for the input `rd` ConfigMap, not `rd-template`. The LimaVM's `DeleteOwnedResources` won't find `rd-template` either, since it has no ownerReference yet. The ConfigMap validator (`controller.go:276`) rejects deleting ConfigMaps that carry the `/owned` finalizer, so `rd-template` becomes undeletable.

The window is narrow (one reconcile cycle), but the result is permanent: an orphan resource that resists manual cleanup.

Fix: Have the App controller's cleanup also delete `rd-template` by deterministic name, or adopt `DeleteOwnedResources` for the entire App cleanup path (it would handle both the input ConfigMap and the LimaVM in one pass).

1. **`ObservedGeneration` forwarded from LimaVM context to App** — `pkg/controllers/app/app/controllers/app_controller.go:181` [Claude Opus 4.6, Gemini 3.1 Pro]

### Design Observations

1. **The deletion path should use `DeleteOwnedResources` instead of manual find-and-delete** (in-scope)

The App controller manually finds the LimaVM by name and deletes it. This bypasses the established pattern where `DeleteOwnedResources` strips `/owned` finalizers and deletes owned resources in one pass. Adopting `DeleteOwnedResources` for App deletion eliminates the `/owned` deadlock (Critical #1), handles the `rd-template` orphan race (Important #2) via transitive ownership, and reduces the code to a single function call. It also means any future owned resources are automatically covered without modifying the deletion path.

2. **Bootstrap handoff through two ConfigMaps creates avoidable lifecycle coupling** (future)

The current flow creates an input `rd` ConfigMap, has the LimaVM webhook copy it into `rd-template`, then has the reconciler adopt `rd-template`. This split ownership creates a window where `rd-template` has no owner. A simpler design would have the App controller create the final template directly, or have the LimaVM webhook set the ownerReference atomically. Either approach eliminates the orphan window and makes deletion semantics easier to reason about.

3. **Embedded template is a temporary bootstrapping mechanism** (future)

The template is embedded via `//go:embed lima.yaml` and marked temporary (`controller.go:18`). When this moves to a proper template source (e.g., the TwoCows project per issue #227), the `LimaTemplateData string` field will need to support runtime resolution (e.g., host architecture selection, dynamic URLs). The current string injection design makes this transition straightforward.

```bash
    rdd ctl wait --for=jsonpath='{.spec.running}'=true \
        limavm/"${VM_NAME}" --namespace "${RDD_NAMESPACE}" --timeout=30s
}
```

The new tests verify that `Created=True` appears and that App patches reach `LimaVM.spec.running` at `bats/tests/32-app-controllers/app.bats:114-151`, but they never wait for the LimaVM `Running` condition or for the App to mirror `Running`. That means the suite would not catch the architecture bug above, nor any breakage in the actual start/stop lifecycle; it only proves metadata propagation.

Fix: extend the bats flow to set `running: true`, wait for `limavm.status.conditions[type=Running].status=True`, and verify the App mirrors that condition back. Add a fast delete-after-create case as well to cover the orphan race.

### Design Observations

1. **The bootstrap handoff through two ConfigMaps creates avoidable lifecycle coupling** (in-scope)

The current flow creates a temporary `rd` ConfigMap in the App controller, asks the LimaVM webhook to copy it into `rd-template`, then asks the LimaVM reconciler to adopt `rd-template` later. That split ownership is what creates the orphan window above. A simpler design is to have the App controller render the final template directly once it knows the host facts, or to have LimaVM cleanup always own deletion of `GetTemplateConfigMapName()` by name even before owner adoption. Either approach removes a class of timing bugs and makes deletion semantics much easier to reason about.

### Testing Assessment

1. No test deletes the App immediately after LimaVM creation but before the LimaVM reconciler sets the owner reference on `rd-template`; that is the highest-risk missing scenario because it can strand an undeletable resource.
2. No test covers host-dependent template selection across `amd64` and `arm64`, even though the feature requirement explicitly depends on host architecture/image choice.
3. The `running=true` / `running=false` tests only assert `LimaVM.spec.running`, not the actual `Running` condition on LimaVM or App, so they do not validate the real start/stop lifecycle.

### Documentation Assessment

The PR description and code comments still imply host-specific template selection from issue #227, but the implementation is currently a single embedded template. When this is fixed, document the selection rules explicitly and record which controller is responsible for deleting the final template ConfigMap during early-delete races.

### Gemini 3.1 Pro

### Executive Summary

This PR implements the core `App` controller reconciliation loop, successfully managing the creation, state propagation, and teardown of the owned `LimaVM` singleton. However, the PR contains critical regressions: it permanently deadlocks the deletion of the `App` resource by failing to remove a blocking finalizer from the child `LimaVM`, and it entirely misses the architectural requirement to dynamically select the correct host architecture (e.g., `amd64` vs `arm64`), breaking support for Apple Silicon. Additionally, there are several edge cases where transient API errors are swallowed, leading to potential resource leaks, and observable generation numbers are mirrored incorrectly.

### Critical Issues

1. **LimaVM stuck in Terminating state forever during App deletion** — `pkg/controllers/app/app/controllers/app_controller.go:74-78` `(critical, regression)`

```go
                case !base.IsBeingDeleted(limaVM):
                        if err := r.Delete(ctx, limaVM); err != nil && !apierrors.IsNotFound(err) {
                                return ctrl.Result{}, fmt.Errorf("failed to delete LimaVM: %w", err)
                        }
                        log.Info("Requested LimaVM deletion, waiting for teardown")
                        return ctrl.Result{RequeueAfter: requeueAfterDeletion}, nil
```

The `App` controller sets the `base.OwnedFinalizerName` (`rdd.rancherdesktop.io/owned`) finalizer on the `LimaVM` at creation to block unmanaged cascade deletion. During teardown, the controller issues `r.Delete(ctx, limaVM)` and polls until the resource is gone. However, it never strips the `OwnedFinalizerName` from the `LimaVM`. As a result, the `LimaVM` becomes permanently stuck in the `Terminating` state, the App controller loops infinitely, and the `App` object hangs indefinitely on deletion (timing out operations like `rdd ctl delete app`).

Fix: Explicitly remove the cascade-blocking finalizer before or alongside issuing the delete:
```go
                case !base.IsBeingDeleted(limaVM):
                        if err := base.RemoveOwnedFinalizer(ctx, r.Client, limaVM); err != nil {
                                return ctrl.Result{}, fmt.Errorf("failed to remove owned finalizer: %w", err)
                        }
                        // ... then delete
```

2. **Hardcoded architecture and image format breaks Apple Silicon support** — `pkg/controllers/app/app/lima.yaml:3` and `pkg/controllers/app/app/controllers/app_controller.go:116-118` `(critical, gap)`

```yaml
images:
- location: https://github.com/rancher-sandbox/rancher-desktop-opensuse/releases/download/v0.1.0/distro.v0.1.0.amd64.qcow2
```

The linked issue explicitly requires the controller to "Select correct arch (amd64/arm64) and image format (qcow2/tar) from host". This PR embeds a static `lima.yaml` that hardcodes the image to `distro.v0.1.0.amd64.qcow2` and makes no attempt to substitute the value during ConfigMap creation. Since the template also disables Rosetta (`rosetta: enabled: false`), booting this configuration on Apple Silicon (M1/M2/M3, which are `aarch64`) will fail entirely or degrade to unusably slow QEMU instruction translation.

Fix: Use `text/template` or `strings.ReplaceAll` to dynamically substitute `amd64.qcow2` with the correct architecture string (e.g. via `runtime.GOARCH`) before writing `r.LimaTemplateData` to the ConfigMap.

### Important Issues

1. **Incorrect `ObservedGeneration` mapping corrupts status conditions** — `pkg/controllers/app/app/controllers/app_controller.go:181` `(important, regression)`

```go
               statusChanged = apimeta.SetStatusCondition(&app.Status.Conditions, metav1.Condition{
                       // ...
                       Message:            cond.Message,
                       ObservedGeneration: cond.ObservedGeneration,
                       LastTransitionTime: cond.LastTransitionTime,
               }) || statusChanged
```

The reconciler copies the `ObservedGeneration` directly from the child `LimaVM`'s conditions onto the parent `App`'s conditions. This is an anti-pattern. `ObservedGeneration` on a status condition must reflect the generation of the object hosting the status (the `App`), not the generation of the child resource. If `App` and `LimaVM` generation numbers ever diverge (e.g., a metadata or unrelated spec update increments `App`'s generation but not `LimaVM`'s), consumers will see a mismatched `ObservedGeneration` and incorrectly assume the `App` controller has permanently stalled on an older spec.

Fix: Change this line to `ObservedGeneration: app.Generation`.

2. **Swallowed API errors during ConfigMap creation cause permanent leak** — `pkg/controllers/app/app/controllers/app_controller.go:163-165` `(important, regression)`

```go
       } else if !apierrors.IsNotFound(err) {
               log.Error(err, "Failed to fetch input ConfigMap")
       }
```

If `r.Get` fails with a transient error (e.g., API timeout or rate-limiting) when trying to clean up the bootstrap ConfigMap, the controller logs the error and proceeds to update the status, ultimately returning `ctrl.Result{}, nil`. Because the App controller does not Watch `ConfigMap` resources, it will never be requeued to retry this deletion unless an unrelated spec/status change triggers another reconcile. This quietly leaks the bootstrap ConfigMap for the entire lifetime of the `App` object.

Fix: Return `ctrl.Result{}, err` to ensure the controller requeues and retries the cleanup.

3. **Swallowed API errors during teardown prematurely drop App finalizer** — `pkg/controllers/app/app/controllers/app_controller.go:68-71` `(important, regression)`

```go
                        } else if !apierrors.IsNotFound(cmErr) {
                                log.Error(cmErr, "Failed to fetch input ConfigMap during cleanup")
                        }
                        return ctrl.Result{}, base.RemoveCleanupFinalizer(ctx, r.Client, &app)
```

Similar to the leak above, if `r.Get` fails with a transient error during App deletion, the error is swallowed and the controller proceeds to unconditionally remove the `App`'s cleanup finalizer. While Kubernetes GC will eventually clean up the ConfigMap in the background since it has an OwnerReference, this circumvents the controller's intended design of synchronously verifying cleanup before unblocking the `App`'s deletion.

Fix: Return `ctrl.Result{}, cmErr` instead of proceeding to remove the finalizer if `cmErr` is not `IsNotFound`.

4. **Dropped Kubernetes Events degrades observability** — `pkg/controllers/app/app/controllers/app_controller.go:195-206` `(important, regression)`

The PR completely removes the `Recorder: events.EventRecorder` from the `AppReconciler` struct and deletes the `setCondition` helper that previously emitted standard Kubernetes Events (e.g., `ConditionChanged`) when the `App` status changed. Since `App` is a top-level singleton resource heavily monitored by administrators/users, silently dropping these events significantly degrades cluster observability and debugging capabilities.

Fix: Restore the `Recorder` in the reconciler struct and emit a `corev1.EventTypeNormal` event whenever `statusChanged` evaluates to true after mirroring conditions.

### Suggestions

1. **Unnecessary active polling during LimaVM teardown** — `pkg/controllers/app/app/controllers/app_controller.go:78-82` `(suggestion, enhancement)`

```go
                        return ctrl.Result{RequeueAfter: requeueAfterDeletion}, nil
                default:
                        // LimaVM deletion is in progress; keep requeueing until it is gone.
                        log.Info("Waiting for LimaVM deletion to complete")
                        return ctrl.Result{RequeueAfter: requeueAfterDeletion}, nil
```

The App controller uses a 2-second `RequeueAfter` loop while waiting for the `LimaVM` to be fully deleted. However, the reconciler has `Owns(&limav1alpha1.LimaVM{})` configured in its `SetupWithManager`. The controller-runtime machinery will automatically and instantaneously wake the reconciler the moment the `LimaVM`'s finalizer is stripped and it is deleted from the API server. Using `RequeueAfter` forces redundant active polling in an otherwise purely event-driven architecture.

Fix: Return `ctrl.Result{}, nil` and rely on the watch event to immediately trigger the final cleanup step.

2. **Dangling conditions on App if dropped from LimaVM** `(suggestion, gap)`

`apimeta.SetStatusCondition` exclusively updates or appends conditions. The condition-mirroring loop does not prune old conditions. If the `LimaVM` controller ever drops a condition type entirely rather than transitioning it to `False`, the `App` resource will retain the stale, orphaned condition indefinitely.
